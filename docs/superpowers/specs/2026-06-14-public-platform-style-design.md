# 面向公众平台风格设计方案

## 1. 背景与目标

当前前端整体视觉与布局以内部工作台、后台管理系统为中心，适合研发协作与系统操作，但不适合未来面向公众的平台形态。后续平台除了继续承载内部协作，还计划逐步发展团队 / 企业订阅模式，因此需要一套同时满足以下目标的公众端设计基线：

- 对外呈现开放、可参与、可信赖的平台形象，而不是后台系统外放。
- 支持未登录用户浏览社区、项目、动态与活动，形成增长入口。
- 支持已登录用户快速回到项目、协作和继续执行的工作上下文。
- 为团队 / 企业订阅做好产品叙事和视觉承接，而不是后期再临时补商业化包装。

本设计方案聚焦公众平台的视觉方向、首页信息架构、关键组件范式、商业适配方式以及与现有后台的共存改造路径。

## 2. 产品定位

建议将未来平台统一定义为三层结构：

### 2.1 Public Community

公开社区层，面向未登录用户开放。主要职责是：

- 展示平台正在发生的真实协作与内容流动。
- 展示精选项目、案例、专题、活动与社区成员。
- 建立“这是一个真实、活跃、值得参与的平台”的第一印象。

这一层的目标是增长、认知和信任，不以直接销售为主。

### 2.2 Team Workspace

协作工作区层，面向已登录个人与团队用户。主要职责是：

- 让用户快速回到我的项目、推荐项目、最近继续事项。
- 承接项目创建、协作加入、基础 AI 能力与日常工作入口。
- 作为免费试用或基础版本的核心使用层。

这一层的目标是使用承接与协作留存。

### 2.3 Enterprise Control

企业治理层，作为团队 / 企业订阅的核心价值承载层。主要职责是：

- 私有空间、组织管理、权限治理、审计与合规。
- 更高资源额度、高级 AI 能力、企业集成、部署与 SLA。
- 支撑团队规模化协作，而不是单纯卖浏览资格。

这一层的目标是商业化转化。

## 3. 总体设计方向

### 3.1 风格路线

建议采用 `Open Commons` 风格路线。

该路线保留 GitHub 风格的秩序感、可读性和结构理性，但从纯工程工具审美调整为更温润、更开放、更欢迎参与的社区平台气质。

### 3.2 核心关键词

- 开放
- 协作
- 温和可信
- 克制清晰
- 长期可读

### 3.3 风格判断标准

如果设计方向正确，用户打开公众端后应优先感受到：

- 这里有人在协作，而不是一套管理后台。
- 这里有持续更新的项目和社区，而不是静态宣传页。
- 这里适合团队长期使用，而不是一次性营销落地页。

## 4. 首页信息架构

公众平台首页采用“同一骨架、按登录状态切换重心”的策略，避免做成两套完全割裂的产品。

### 4.1 未登录首页

未登录首页主位以社区动态为中心。

建议结构如下：

1. 轻量品牌头部：品牌名、核心导航、登录、加入入口。
2. 首屏英雄区：左侧为平台价值主张、当前热门主题与加入入口，右侧为社区动态预览。
3. 精选项目带：展示值得浏览与信任背书的项目或案例。
4. 社区讨论 / 动态带：展示最近讨论、更新、协作行为。
5. 活动 / 专题带：展示主题活动、精选专题、合作计划。
6. 加入社区带：引导注册、加入团队、开始协作。

未登录首页的首要目标不是立即卖订阅，而是建立“社区正在发生”的感知。

### 4.2 已登录首页

已登录首页主位切换为项目展示与继续协作入口。

建议结构如下：

1. 轻量品牌头部：保留与公众端一致的品牌结构，同时增加个人入口。
2. 首屏主位：我的项目、推荐项目、最近继续事项。
3. 快速入口区：创建项目、加入项目、查看待处理事项。
4. 社区动态带：保留社区流，作为辅助层，确保登录后仍能感知公共平台活力。
5. 推荐内容 / 精选专题带：作为发现型内容补充。

已登录首页的目标是让用户快速进入协作场景，而不是重新回到传统后台仪表盘。

## 5. 视觉系统建议

### 5.1 色彩方向

建议以暖白与石墨灰为基础，使用温和的自然色系作为品牌强调。

建议色彩角色如下：

- 页面底色：暖白、米灰白，避免纯白造成过冷观感。
- 卡片底色：云白或低对比浅白，提高内容承载稳定性。
- 主文字：深石墨灰，保证阅读性。
- 边界线：低对比灰褐色或浅灰线，减弱后台工具味。
- 主强调色：苔绿色或温暖铜棕。
- 信息辅助色：灰蓝或低饱和蓝绿色，用于信息状态与中性辅助。

不建议继续沿用过强的内部工作台操作色逻辑，也不建议使用高饱和科技蓝或常见 AI 风格紫白渐变。

### 5.2 字体方向

字体系统要从“系统工具感”转向“社区平台可读感”。

建议原则如下：

- 标题字体应克制、有识别度，但不能夸张到影响技术平台可信度。
- 正文字体优先高可读、长时间阅读舒适。
- 中文与西文字体搭配要兼顾工程感和温和感。
- 标题权重高于正文字重，形成更清晰的内容层级。

### 5.3 布局气质

公众端布局应更像网站与社区平台，而不是工作台控制台。

建议原则如下：

- 使用顶部导航，而不是左侧后台功能树。
- 使用清晰的内容分区与大面积留白。
- 使用卡片、内容流、标签与头像组表达“人”和“项目”的存在。
- 控制装饰性背景，只做轻微氛围，不抢内容本身。

## 6. 关键页面范式

### 6.1 公众首页

公众首页应是“社区叙事 + 动态预览 + 项目展示”的综合门户，而不是企业官网或后台首页。

核心特征：

- 首屏传达平台价值与活跃度。
- 中段展示精选项目与案例。
- 下段展示讨论、动态、活动与加入入口。

### 6.2 项目发现页

项目页不应直接复用后台表格思维，而应采用“项目卡 + 紧凑列表可切换”的方式。

每个项目单元建议包含：

- 项目名称
- 项目简述
- 标签
- 负责人 / 成员信息
- 最近活跃时间
- 活跃度或参与度
- 进入项目入口

### 6.3 社区动态页

动态页应采用 Feed 结构，而不是通知中心结构。

动态内容建议包括：

- 项目更新
- 成员讨论
- 活动开始
- 专题发布
- 协作招募

每条动态都应具备来源对象、动作、摘要和跳转目的地。

### 6.4 已登录首页

已登录首页应作为协作门户存在，而不是传统统计看板。

主位应强调：

- 我的项目
- 推荐项目
- 最近继续事项
- 快速创建 / 加入 / 查看待处理入口

### 6.5 定价与企业方案页

商业页应从成长路径出发，而不是单纯价格表。

建议按以下层级组织：

- 个人探索
- 团队协作
- 企业治理

这样更符合团队 / 企业订阅的转化逻辑。

## 7. 关键组件语义

公众端建议优先沉淀以下语义组件：

### 7.1 项目卡

平台最核心的内容单元，用于项目发现、精选项目、推荐项目等场景。

### 7.2 动态卡

用于承载项目更新、社区讨论、活动通知与专题内容，形成统一社区 Feed 语义。

### 7.3 标签系统

统一承载项目方向、技术栈、难度、状态、招募、专题等信息表达。

### 7.4 成员头像组

持续表达“平台里有人”，弱化纯系统工具感。

### 7.5 快速入口条

承载加入社区、创建项目、参与讨论、查看活动等高频入口。

### 7.6 专题横幅

承载阶段性活动、精选主题、合作计划或运营位，不作为首页长期唯一视觉主角，但需要具备拉升品牌气氛的能力。

### 7.7 输入框与窄容器降级规范

公众端带前置图标的搜索输入框统一使用 `frontend-public/src/components/common/Input.tsx`，不要在页面中重复实现绝对定位图标、左内边距和 focus 样式。

通用 `Input` 的默认样式必须保持兼容；只有筛选栏、搜索栏等明确需要适配窄容器的场景才开启 `adaptiveIcon`，并按需要使用 `size="sm"` 和 `wrapperClassName` 控制布局：

```tsx
<Input
  size="sm"
  adaptiveIcon
  wrapperClassName="min-w-0 max-w-xs flex-1"
  icon={<Search className="h-4 w-4" />}
  placeholder="搜索…"
  value={keyword}
  onChange={(event) => setKeyword(event.target.value)}
/>
```

自适应规则由通用组件内部的容器查询负责：容器宽度足够时保留常规 focus ring；容器小于 280px 时增加图标与文字的左间距，并关闭外层 ring，避免搜索文字被图标或焦点轮廓遮挡。需要额外操作按钮（例如清空）的场景，应在外层保留 `relative` 容器，按钮与 `Input` 并列放置。

带搜索建议或搜索结果的输入框，结果面板必须放在同一个 `relative` 容器内，并使用 `absolute left-0 right-0 top-full` 定位，配合 `z-30`、边框、背景色和最大高度滚动。结果面板不参与正常文档流，避免把目录树或列表顶下去；除非产品明确要求，否则不使用阴影。

该规范已应用于发布与观测、测试与执行、文档 Wiki、API 搜索和知识图谱搜索。普通表单输入默认不启用自适应能力，以避免无关页面发生视觉变化。

## 8. 商业化适配原则

既然未来收费模式偏向团队 / 企业订阅，则公众平台与商业化之间应遵循以下原则：

### 8.1 社区负责获客与信任

公开内容、项目展示、动态流和活动不是商业化障碍，反而是建立信任和增长的关键。

### 8.2 协作负责承接试用

用户登录后应能快速进入协作，而不是先看到重营销转化页。

### 8.3 订阅卖治理与规模化能力

付费重点建议放在：

- 私有项目 / 私有空间
- 组织与权限管理
- 审计与合规
- 更高额度与高级能力
- 企业集成与部署
- SLA 与服务保障

### 8.4 避免的风险

需要避免以下两类错误：

- 表面做社区，实际全部流量导向硬性销售，破坏社区可信度。
- 过早限制基础参与能力，导致平台既不像社区，也不利于公开内容沉淀。

## 9. 与现有前端的改造关系

### 9.1 布局分离

现有 [frontend/src/layout/AppLayout.vue](../../frontend/src/layout/AppLayout.vue) 明显服务于后台工作台场景，不适合作为公众端骨架。

建议新增 `PublicLayout`，用于公众平台页面；后台继续保留现有工作台布局。

### 9.2 主题分层

现有 [frontend/src/styles/index.css](../../frontend/src/styles/index.css) 已具备统一设计变量基础，但当前整体偏向后台工作台。

建议拆成：

- Base Tokens：字体、圆角、阴影、基础状态色、通用间距。
- Console Theme：后台工作台主题。
- Public Theme：公众平台主题。

这样既能复用基础能力，又能避免公众端与后台端相互污染。

### 9.3 页面优先级

第一阶段不建议全站一次性翻新，优先做以下页面：

1. 未登录首页
2. 已登录首页
3. 项目发现页

这三页最能快速建立公众平台新气质，也最容易验证风格方向是否正确。

### 9.4 交互模式迁移

公众端应从“表格操作思维”迁移到“卡片 + 内容流 + 入口条思维”。

后台端继续保留管理效率优先的高密度操作结构，两端通过统一品牌和基础设计语义保持同源。

## 10. 建议的实施顺序

### 10.1 第一阶段：建立公众端骨架

- 新增 `PublicLayout`
- 建立公众端导航、页脚、公共内容区结构
- 定义 Public Theme 变量

### 10.2 第二阶段：实现核心页面

- 未登录首页
- 已登录首页
- 项目发现页

### 10.3 第三阶段：扩展社区能力

- 动态 / 讨论页
- 活动 / 专题页
- 定价页 / 企业方案页

### 10.4 第四阶段：统一品牌体验

- 调整组件语义
- 沉淀公众端组件库
- 明确后台与公众端的跨端品牌一致性

## 11. 最终结论

本方案建议未来平台采用“开放社区获取信任，登录协作承接使用，团队治理完成订阅转化”的整体路径。

在设计上，公众端采用 `Open Commons` 风格路线，以 GitHub 式秩序感为基础，强化开放、温和、可参与的社区气质；在产品结构上，未登录首页以社区动态为主，已登录首页以项目展示与继续协作为主；在商业承接上，将团队 / 企业订阅能力聚焦在治理、私有化与规模化协作能力，而不是对社区内容本身收费。

该路线既能解决现有后台气质不适合对外的问题，也能为未来团队 / 企业订阅模式提供自然、可信且长期稳定的产品表达基础。
