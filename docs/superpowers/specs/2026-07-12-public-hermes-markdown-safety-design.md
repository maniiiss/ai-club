# 公众端 Hermes Markdown 安全渲染设计

## 目标

保证 Hermes 助手和聊天室在接收到合法 Markdown 时原样交给 `react-markdown` 渲染，不再由前端归一化逻辑把行内项目编号或成对强调标记改坏。

## 已确认事实

- Hermes 独立会话消息 `id=436` 在数据库中的原文包含 `项目 #4` 与 `**【PC端】审批台帐中列表个别字段没有回显**`，语法合法。
- 该原文经过 `normalizeGeneratedMarkdown` 后被拆成 ATX 标题、未闭合强调标记和空列表项，能够与截图逐字复现。
- 聊天室消息列表在外层与 `Markdown` 组件内各调用一次归一化器，存在重复处理。

## 方案

1. 为 `Markdown` 增加显式的 `normalize` 开关，默认保留现有行为，避免影响其他页面。
2. Hermes 独立会话传入 `normalize={false}`，将后端助手原文直接交给 `react-markdown`。
3. 聊天室只保留既有的外层归一化调用，并为 `Markdown` 传入 `normalize={false}`，消除第二次处理。
4. 为真实 Hermes 原文添加回归测试，验证合法行内 `#4` 和成对 `**...**` 在禁用归一化的助手路径中保持不变；同时保留既有归一化测试，确保其其他消费者不发生未预期变更。

## 非目标

- 不继续向 `markdownUtils.ts` 添加新的正则修复规则。
- 不修改 Hermes 模型、服务端请求或数据库中的历史消息。
- 不改变非助手页面的 Markdown 归一化行为。

## 验证

- 定向运行 Markdown 及渲染入口测试。
- 运行公众端完整测试与构建。
- 运行项目编码检查。
