# 公众 SaaS 产品前端技术设计 v1

## 1. 背景

AI Club 当前主前端位于 `frontend/`，定位是 Vue 3 + Element Plus 的私有化管理控制台，核心页面围绕平台管理员、项目负责人、研发/测试/DevOps 等内部角色设计。它覆盖项目、迭代、工作项、Wiki、API、GitLab、执行中心、测试、流水线、可观测性、服务器、模型和系统治理等完整能力。

仓库中已经存在 `frontend-public/`，技术栈为 React + Vite + Tailwind，但当前更像一份概念演示原型：应用名仍是模板名，页面内容是 Aether R&D Dashboard，数据主要来自本地 mock，尚未形成真实产品前端。

本设计面向“任何人可注册使用的 SaaS 版”第一阶段。当前阶段暂不引入多租户、计费和复杂组织隔离，优先把产品前端体验完整做出来，并通过现有用户、权限和项目数据模型裁剪出面向公众用户的单工作区 SaaS 使用路径。

## 2. 目标与非目标

### 2.1 目标

- 将 `frontend-public/` 升级为 AI Club 公众 SaaS 产品前端，而不是现有控制台换肤。
- 围绕“规划、开发、测试、发布、观测、复盘”的项目闭环重组核心功能入口。
- 第一阶段复用现有后端、登录态、权限、项目、迭代、执行、测试、GitLab、流水线、可观测性等 API。
- 暂不做多租户，通过默认角色、功能开关和权限裁剪控制普通 SaaS 用户可见能力。
- 保留 `frontend/` 作为私有化管理控制台，避免影响当前交付链路。
- 为后续多租户、计费、额度、组织管理预留前端信息架构扩展点。

### 2.2 非目标

- 不在第一阶段重构后端数据模型为租户级模型。
- 不开放服务器 SSH/SFTP、环境变量管理、平台工具配置、自升级中心等高风险后台能力给普通用户。
- 不把现有 `frontend/` 的所有页面逐个复制到 `frontend-public/`。
- 不在第一阶段实现计费、套餐、额度扣减和支付流程。
- 不替代私有化部署版本；两个前端长期并存。

## 3. 影响范围

- `frontend-public/`
  - 从演示原型升级为新的 SaaS 产品前端。
  - 新增真实路由、布局、API client、认证状态、权限裁剪、空状态、错误状态和移动端适配。
- `frontend/`
  - 保持当前私有化管理控制台定位。
  - 可作为 API 类型、页面链路和复杂交互的参考来源，不直接迁移样式体系。
- `backend/`
  - 第一阶段只做最小适配：注册默认角色、运行时能力配置、必要聚合接口和权限裁剪。
  - 高风险接口继续依赖现有权限校验。
- `docs/`
  - 本设计作为公众 SaaS 前端基线方案，后续实现计划和架构变更以此为输入。

## 4. 现状与问题分析

### 4.1 现有 `frontend/` 的特点

现有主前端是成熟的管理控制台，优势是业务覆盖完整、权限链路清晰、与后端 API 对齐充分。但它的问题也很明显：

- 页面组织以后台菜单为核心，不符合公众 SaaS 用户的首次使用路径。
- Element Plus 表格、弹窗、管理列表密度较高，面向内部管理者效率优先，不适合公众产品体验。
- 系统级配置、服务器管理、模型管理、环境变量等入口与项目协作能力混在同一套导航中，公众用户理解成本高。
- 直接在该前端改造容易影响私有化版本稳定性。

### 4.2 现有 `frontend-public/` 的特点

`frontend-public/` 技术栈更适合作为新产品前端起点：

- React + Vite + Tailwind 更适合快速构建独立 SaaS 产品界面。
- 与 `frontend/` 解耦，可大胆调整视觉、导航和交互。
- 当前构建与类型检查可通过，具备继续演进基础。

它的不足是：

- 缺少真实路由和 API client。
- 缺少登录、注册、权限、全局错误处理和运行时配置。
- 页面内容仍是本地假数据，且偏研发仪表盘演示，不是 AI Club 的真实产品信息架构。
- 没有和现有后端核心模块建立契约。

## 5. 设计方案

### 5.1 总体方案

采用“双前端并存”策略：

```text
用户浏览器
  -> frontend-public：公众 SaaS 产品前端
  -> backend：复用现有业务 API、登录态、权限校验

管理员 / 私有化客户浏览器
  -> frontend：私有化管理控制台
  -> backend：现有完整后台能力
```

`frontend-public` 负责重新组织 SaaS 产品体验，`backend` 继续作为统一业务后端。第一阶段不新增租户隔离表，不改变现有项目主数据归属方式，只通过默认用户角色、权限配置和功能开关控制可见能力。

### 5.2 产品信息架构

公众 SaaS 版按用户任务组织为 7 个一级区：

| 一级区 | 目标 | 覆盖能力 |
|---|---|---|
| 首页 | 让用户知道下一步该做什么 | 项目进展、今日待办、最近执行、AI 建议、快捷入口 |
| 项目 | 以项目空间承载协作上下文 | 项目列表、项目详情、成员摘要、进度、风险、活动、分享 |
| 计划 | 管理需求、任务、缺陷与迭代节奏 | 迭代、工作项、看板、列表、工作项详情、AI 拆解 |
| 知识 | 沉淀项目文档与上下文 | Wiki、项目文档、记忆事实图、项目问答 |
| 研发 | 连接代码仓库与质量治理 | 仓库绑定、代码结构、MR 辅助、扫描、自动合并摘要 |
| 测试与执行 | 跟踪测试和异步任务 | 测试计划、测试用例、执行中心、自动化结果、产物报告 |
| 发布与观测 | 关注发布是否成功、线上是否健康 | 流水线、发布记录、运行历史、健康趋势、日志摘要 |

设置区收纳低频能力：

- 个人资料
- GitLab 授权
- 通知偏好
- 模型使用信息
- 管理员可见的基础配置入口

第一阶段不在公众导航中展示：

- 用户管理、角色管理、权限管理
- 环境变量管理
- 工具配置
- 服务器 SSH/SFTP
- Jenkins 服务实例管理
- 自升级中心
- 平台级扫描规则集管理
- 操作日志全局查询

### 5.4 路由设计

第一阶段建议路由：

| 路由 | 页面 |
|---|---|
| `/login` | 登录 |
| `/register` | 注册 |
| `/` | SaaS 首页，登录后重定向到 `/dashboard` |
| `/dashboard` | 工作台 |
| `/projects` | 项目列表 |
| `/projects/:projectId` | 项目详情 |
| `/projects/:projectId/planning` | 迭代与工作项 |
| `/projects/:projectId/knowledge` | 知识库 |
| `/projects/:projectId/development` | 研发链路 |
| `/projects/:projectId/execution` | 测试与执行 |
| `/projects/:projectId/release` | 发布与观测 |
| `/settings/profile` | 个人设置 |
| `/settings/integrations` | 外部授权 |
| `/public/projects/:projectId/readonly/:token` | 项目公开只读页 |

项目详情下采用项目内二级导航，避免全局导航过长。

### 5.5 核心页面设计

#### 首页

首页是行动中心，不是后台指标堆叠。建议包含：

- 最近访问项目
- 今日待处理工作项
- 最近执行结果
- 发布/测试/观测异常提醒
- Hermes AI 建议
- 快速创建项目、导入仓库、新建任务

#### 项目详情

项目详情是 SaaS 版主战场。建议顶部固定展示：

- 项目名称、状态、负责人
- 当前迭代
- 工作项完成度
- 测试状态
- 最近发布状态
- 健康状态

下方用模块卡片进入计划、知识、研发、测试与执行、发布与观测。

#### 计划

计划区同时支持看板和列表：

- 看板适合普通用户理解状态流转。
- 列表适合批量查看需求、任务、缺陷。
- 工作项详情使用抽屉或独立详情页。
- AI 动作以“生成验收标准”“拆解子任务”“生成测试点”等建议型能力出现。

#### 知识

知识区聚合 Wiki 和项目上下文：

- Wiki 空间与页面树。
- 最近更新文档。
- 项目记忆事实图入口。
- Hermes 项目问答入口。

#### 研发

研发区面向项目成员，而不是平台管理员：

- 连接仓库。
- 查看代码结构。
- 查看 MR / 自动合并摘要。
- 发起质量扫描。
- 展示扫描结果和风险趋势。

底层 GitLab token、产品分支、Webhook 等复杂配置尽量后置到高级设置。

#### 测试与执行

测试与执行区强调结果可追踪：

- 测试计划列表。
- 测试用例摘要。
- 自动化执行状态。
- 执行中心任务列表。
- 执行详情、报告、截图、Trace、产物入口。

#### 发布与观测

发布与观测区回答三个问题：

- 这次发布成功了吗？
- 当前线上健康吗？
- 有哪些异常需要处理？

主要展示：

- 流水线摘要。
- 发布/运行历史。
- 项目健康趋势。
- 应用日志摘要。
- 项目公开分享入口。

### 5.6 认证与权限

第一阶段复用现有登录态和权限校验：

- `frontend-public` 维护独立 auth store，保存当前用户、权限摘要和登录状态。
- 路由进入前拉取用户 profile。
- 无登录态跳转 `/login`。
- 无权限页面显示 SaaS 风格的能力不可用提示，不直接暴露后台 403 文案。

建议后端补充默认 SaaS 用户角色：

- `SAAS_USER`：普通注册用户，拥有项目协作主链路权限。
- `SAAS_ADMIN`：当前单工作区管理员，拥有部分配置权限。

第一阶段普通用户不授予系统级配置权限。

### 5.7 功能开关

新增运行时能力配置接口，供 `frontend-public` 判断哪些能力可显示：

```text
GET /api/public-runtime-config
```

建议返回：

```json
{
  "registrationEnabled": true,
  "gitlabIntegrationEnabled": true,
  "cicdEnabled": true,
  "observabilityEnabled": true,
  "serverManagementVisible": false,
  "modelProviderManagementVisible": false,
  "selfUpgradeVisible": false
}
```

第一阶段如果暂不新增接口，也可以先在前端配置静态 feature flags，但正式接入前应收口到后端运行时配置。

### 5.8 API 接入策略

`frontend-public` 不直接复用 `frontend/src/api` 文件，避免跨框架路径和依赖耦合。建议重新建立 React 侧 API client，但字段和接口契约参考现有实现：

- `frontend/src/api/platform.ts`
- `frontend/src/api/gitlab.ts`
- `frontend/src/api/cicd.ts`
- `frontend/src/api/observability.ts`
- `frontend/src/api/wiki.ts`

API client 应统一处理：

- baseURL
- 登录 token
- 401 跳转
- 后端 `ApiResponse<T>` 解包
- 错误消息归一化
- 请求取消和 loading 状态

### 5.9 视觉与交互方向

公众 SaaS 版采用清爽、可信、低噪声的产品协作工具风格：

- 明亮背景，减少后台系统的灰蓝表格感。
- 保留蓝色品牌，但降低全站蓝色占比。
- 状态色清晰：绿色成功、琥珀风险、红色失败、灰色中性。
- 少用大面积渐变，避免营销页或演示页气质。
- 卡片只用于独立信息单元，不把页面区块全部卡片化。
- 移动端优先保障查看、筛选、轻量评论和状态更新。

AI 能力作为工作辅助呈现，不作为炫技中心：

- 使用建议卡片、动作确认、侧边助手。
- 写操作必须显式确认。
- 失败时说明原因和下一步，而不是只展示技术错误。

## 6. 方案取舍

### 6.1 方案 A：直接改造 `frontend/`

优点：

- 复用最多，短期接入真实 API 最快。
- 权限和复杂页面已经成熟。

缺点：

- 容易变成管理后台换皮。
- 容易影响私有化版本稳定性。
- 很难彻底重做 onboarding、项目空间和公众 SaaS 导航。

结论：不作为长期方案，仅适合短期演示。

### 6.2 方案 B：新建 `frontend-public` SaaS 前端

优点：

- 产品体验可以按 SaaS 心智重建。
- 不影响现有私有化控制台。
- 可逐步接入核心能力，风险可控。
- 后续多租户、计费、组织能力可自然加入。

缺点：

- 需要重新建设 API client、路由、布局和页面。
- 部分复杂业务交互需要二次设计。

结论：推荐采用。

### 6.3 方案 C：React 壳 + 嵌入现有后台页面

优点：

- 演示速度快。
- 能快速覆盖大量功能。

缺点：

- 体验割裂。
- 权限和导航上下文复杂。
- 不适合作为最终 SaaS 产品。

结论：可作为过渡方案，不作为目标形态。

## 7. 风险与兼容性

### 7.1 安全风险

公众 SaaS 用户更容易触达高风险能力。第一阶段必须隐藏或限制：

- 服务器 SSH/SFTP。
- 环境变量管理。
- 平台工具配置。
- 全局用户、角色、权限管理。
- 自升级中心。
- 系统级模型供应商配置。

执行中心、仓库扫描、自动化测试、Hermes 工具调用仍需后端权限和动作确认兜底。

### 7.2 产品范围膨胀

平台模块很多，不能第一版追求全量重写。应先跑通项目闭环，再逐步补高级设置。

### 7.3 双前端契约漂移

两个前端都调用后端 API，可能出现字段和状态口径不一致。应通过文档、类型定义和接口测试控制漂移。

### 7.4 后续多租户迁移

当前阶段不做多租户，但页面命名和数据结构应避免写死“全局唯一工作区”。建议前端保留 `workspace` 概念入口，但第一阶段隐藏组织切换器。

## 10. 待确认问题

- 第一阶段注册是否直接开放，还是仅本地/测试环境开放？
- 普通 SaaS 用户默认能否创建项目，是否需要管理员审核？
- GitLab 集成是否要求用户个人 OAuth，还是允许项目级 token？

- 执行中心是否需要第一阶段限制并发数和单任务时长？
- 模型调用是否使用平台默认模型，是否允许用户自带 API Key？
- 是否需要在 `frontend-public` 中保留私有化控制台跳转入口？
- 第一阶段是否需要部署到独立域名，还是与现有前端同域不同路径部署？
