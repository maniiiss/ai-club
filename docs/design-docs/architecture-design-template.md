# <主题>技术设计

## 1. 背景

说明这次架构调整或技术设计要解决什么问题，当前现状有什么限制，为什么现在要做。

## 2. 目标与非目标

### 2.1 目标

- 明确本次方案必须达成的结果。
- 说明希望改善的指标、体验或工程问题。

### 2.2 非目标

- 写清楚本次不解决什么，避免方案边界不断膨胀。

## 3. 影响范围

- 影响的模块：`frontend` / `backend` / `code-processing` / `scripts` / `docs`
- 影响的链路：页面、接口、异步任务、MCP 工具、部署流程、权限链路等
- 影响的运行配置：端口、环境变量、日志路径、外部依赖

## 4. 现状与问题分析

说明当前实现、现有边界和主要痛点。必要时补充已有链路、瓶颈、失败案例或历史约束。

## 5. 设计方案

### 5.1 总体方案

用文字说明新的架构边界、职责拆分和交互关系。

### 5.2 关键流程

- 入口是什么
- 中间经过哪些服务或模块
- 数据如何流转
- 错误如何处理

### 5.3 数据、接口与配置变更

- 新增或修改的实体、DTO、表结构、迁移脚本
- 新增或修改的接口、事件、任务、工具
- 新增或修改的环境变量、端口、日志和部署要求

## 6. 方案取舍

列出候选方案、为什么没有选、当前方案的收益与代价。

## 7. 风险与兼容性

- 可能影响哪些现有链路
- 如何灰度、回滚或兼容旧逻辑
- 哪些风险需要额外测试或人工确认

## 8. Harness 与验证

- 最小验证 harness
- 扩展验证 harness
- 需要重点关注的日志、监控、回归样例

## 9. 落地计划

- 分阶段实施步骤
- 每一步的交付物
- 哪些步骤依赖前置任务

## 10. 待确认问题

列出仍需评审、业务确认或跨团队协作的问题。
