# 用户定位与个性化公众端工作台技术设计 v1

## 目标

为公众端账号保存一个主定位：项目经理、产品、UI、开发或技术经理。定位用于将首页组织为不同岗位的行动中心，不替代现有角色、权限或项目数据范围。

## 数据与接口

- `user_info.user_position` 使用可空枚举字符串保存。空值表示存量账号未设置定位，公众端继续展示通用工作台。
- 公众端注册请求必须传入定位；管理端用户管理可设置、修改或清空定位。普通用户不能在个人资料中自行修改。
- `/api/auth/login`、`/api/auth/me` 和 `/api/users` 返回定位，保证两个前端可以即时同步展示。
- `/api/dashboard/overview` 在保持原有概览字段的基础上新增 `userPosition` 与 `focusItems`。关注项包含分类、严重度、状态、关联项目/工作项和可选外链。

## 首页聚合规则

所有工作项先通过项目数据权限过滤，再按定位生成最多八条关注项：

| 定位 | 优先内容 |
| --- | --- |
| 项目经理 | 已阻塞、逾期未完成、高优先级未完成工作项，以及个人 MR 合并异常和可见项目的失败构建 |
| 产品 | 需求工作项及其状态、迭代和排期 |
| UI | 本人负责/协作的 UI 设计任务，以及缺少原型链接的需求 |
| 开发 | 本人负责/协作的开发任务和缺陷，以及个人 MR 合并异常、失败构建、快速 MR/构建入口 |
| 技术经理 | 技术设计、开发交付、风险工作项、个人 MR 合并异常和失败构建 |

风险仅根据可验证的工作项状态、计划结束日期和优先级计算：已阻塞或逾期为 `CRITICAL`，未完成高优先级为 `WARNING`。定位不绕过后端鉴权，不返回不可访问项目、任务或仓库的数据。

## 兼容与验证

- Flyway 迁移只新增可空列，不回填或猜测历史岗位。
- 旧服务内部构造器保留兼容重载；外部 HTTP 注册仍由 Bean Validation 强制要求定位。
- 验证覆盖注册校验、用户管理读写与筛选、存量空定位回退、五类首页关注项、项目数据权限过滤，以及两个前端的构建和编码检查。
