# DataWorkbench 语义查询技术设计 v1

## 1. 目标与边界

本设计为现有 DataWorkbench 增加只读的「自然语言数据查询」能力，使用户可以从业务意图获得可解释、可审计的 PostgreSQL 查询结果与总结回复。它与既有 DataChange 共用项目、权限、审计和实体白名单，但不改变 DataChange 的受控写入边界。

v1 的明确边界如下：

- 仅支持动态接入 PostgreSQL 数据源，且连接账户必须为只读账户；MySQL、ClickHouse 等通过后续 Dialect 插件扩展。
- 只支持 `SELECT`，不接受用户、前端或模型提交的原始 SQL；`EXPLAIN ANALYZE`、DML、DDL、`COPY`、多语句和存储过程调用均禁止。
- 模型只参与意图识别、受限 JSON 表达生成和结果总结；语义归一、概念解析、枚举转换、DSL 校验和 SQL 生成均由后端确定性执行。
- 数据源、语义资产和权限策略均按项目隔离；平台管理员可维护全局模板，但不能自动获得业务数据访问权。
- DataChange 继续只面向已绑定的平台项目表的受控更新。跨外部数据源的写操作不在本设计范围内。

## 2. 总体架构与主路径

```text
用户问题
  -> 意图识别（查询 / 澄清 / 拒绝）
  -> 受限表达（槽位、时间、筛选、统计诉求）
  -> 同义词归一（可追溯命中规则）
  -> 概念解析（指标 / 维度 / 实体 / 关系）
  -> 定义展开（口径、粒度、时间语义、权限过滤）
  -> 字典枚举转换（显示词 -> 物理值）
  -> Semantic Query DSL
  -> DSL 校验与 PostgreSQL SQL 编译
  -> 只读执行、脱敏、限流与审计
  -> 基于结果的总结回复
```

后端新增 `data-workbench-query` 应用服务，作为唯一的语义查询编排者。它读取平台库中的数据源注册表、元数据快照与已发布语义版本，通过受控的 PostgreSQL 连接执行参数化查询。`code-processing` 不参与数据源连接；现有 Assistant / Runtime 可复用模型调用能力，但只能得到经过裁剪的候选语义资产和最终结果摘要。

每次查询在开始时固定 `semantic_version_id`、`schema_snapshot_id`、`policy_version` 和 `data_source_id`。因此即使后续修改同义词、指标定义或表结构，历史请求仍可以解释和复现（数据本身变化除外）。

## 3. 语义资产模型

语义资产采用“草稿—校验—发布—废弃”的版本化管理。运行期只读取已发布版本，编辑草稿不影响正在执行或历史查询。

| 资产 | 业务含义 | 关键内容 |
| --- | --- | --- |
| 词条 / 同义词 | 将用户表达归并为稳定术语 | 规范词、同义词、适用项目/数据源、优先级、语言、启停、冲突规则 |
| 概念 | 可被业务理解的对象 | `METRIC`、`DIMENSION`、`ENTITY`、`RELATION`、`FILTER`、`TIME_GRAIN` |
| 定义 | 概念到物理模型的受控映射 | 来源实体、字段/表达式模板、聚合方式、默认时间字段、粒度、可联接关系、示例 |
| 枚举字典 | 人类值与物理值的双向映射 | 字典编码、显示名、物理值、别名、排序、有效期、启停 |
| 权限策略 | 哪些语义资产/数据行/列可被谁访问 | 角色、项目范围、数据源、行级谓词模板、列脱敏规则、最大行数 |

同义词不能直接映射为 SQL 字段。它先映射为规范词，再由概念解析映射为一个或多个候选概念；出现同义词冲突、候选概念并列或缺少必要筛选条件时，服务返回澄清卡片，禁止“猜一个并执行”。

“概念”和“定义”必须分离。例如“新增用户”是指标概念；其定义才规定为 `count(distinct user_id)`、数据源为运营库、时间字段为 `created_at`、统计粒度为日、默认排除测试账号。修改口径会产生新定义版本，而不是覆盖概念本身。

## 4. 动态数据源与元数据发现

### 4.1 数据源生命周期

管理员在「数据工作台 / 数据源」创建数据源时填写名称、项目归属、环境标签、JDBC 地址、数据库/Schema 范围和凭据引用。保存过程必须依次完成：参数校验、网络地址策略校验、只读连接测试、权限检查、Schema 探测、元数据快照生成、管理员确认启用。

凭据只以 AES-GCM 密文或外部密钥引用保存；列表、审计、前端 DTO 和日志一律不返回密码、完整 JDBC URL 或私网主机信息。连接池按 `dataSourceId` 懒加载、数量受限、闲置回收，并在停用、凭据轮换或连接失败阈值触发时销毁。

在“环境变量管理”中维护 `PLATFORM_DATA_WORKBENCH_ALLOWED_DB_HOSTS`（以逗号分隔的 PostgreSQL 主机名或 IP）；该值为空时数据源登记、测试和扫描全部拒绝，避免平台意外成为任意网络探测入口。部署 `.env` 中同名值仅作为后台未配置时的兼容回退。

v1 使用独立 `DataSourceRegistry` 创建 `NamedParameterJdbcTemplate`，不得把动态数据源接入主业务 JPA `DataSource`，以免外部库事务、迁移和连接池影响平台库。

### 4.2 Schema 快照

探测范围限制在登记的 PostgreSQL Schema 与允许表清单内。服务读取 `information_schema` / `pg_catalog`，形成不可变快照：表、列、类型、主键、外键、注释、枚举候选和采集时间。管理员从快照中选择表列创建语义定义，不能手工填写任意物理标识符。

发生列删除、类型变化或数据源不可达时，相关定义标记为 `STALE`；SQL 编译器拒绝使用过期快照。探测任务应由异步任务执行，并记录差异，避免用户查询同步等待元数据刷新。

### 4.3 网络与数据库最小权限

- 平台部署网络需对数据源主机使用显式允许列表；拒绝本机、链路本地、元数据地址及未登记的重定向地址，防止 SSRF。
- PostgreSQL 账户仅授予允许 Schema / 表的 `USAGE` 与 `SELECT`，禁止 `CREATE`、`TEMP`、`EXECUTE`、`SET ROLE` 和写权限。
- 每个查询以只读事务执行，并设置 `SET LOCAL statement_timeout`、`lock_timeout`、`idle_in_transaction_session_timeout`、`default_transaction_read_only=on`；客户端侧同时设置连接与查询超时。
- 数据源级策略配置并由编译器强制附加 `LIMIT`（默认 500、硬上限 2,000）、最大扫描时间和每日/用户配额。

## 5. 查询语言与 SQL 编译

### 5.1 Semantic Query DSL

DSL 是后端内部契约，可在预览界面展示为只读 JSON。它表达业务概念，不包含表名、列名、SQL 片段、函数名或任意标识符。

```json
{
  "version": "1",
  "kind": "QUERY",
  "dataSourceId": 18,
  "semanticVersionId": 42,
  "measures": [{ "concept": "new_user_count", "aggregation": "COUNT_DISTINCT" }],
  "dimensions": [{ "concept": "register_day", "grain": "DAY" }],
  "filters": [
    { "concept": "register_day", "operator": "BETWEEN", "values": ["2026-07-01", "2026-07-07"] },
    { "concept": "user_status", "operator": "IN", "values": ["ACTIVE"] }
  ],
  "orderBy": [{ "field": "register_day", "direction": "ASC" }],
  "limit": 100
}
```

编译器只允许已发布定义中的表、列、关联关系和函数模板；值永远通过命名参数绑定。枚举转换发生在 DSL 校验前：例如“启用、正常用户”会按 `user_status` 字典转换为 `ACTIVE`。未知枚举值、跨数据源关联、未声明关系、无时间范围的大表明细查询，均返回澄清或拒绝，不生成 SQL。

SQL 编译输出为 `CompiledQuery`：参数化 SQL、参数类型、展示用 SQL 摘要、涉及概念/定义、风险标签和预估限制。展示摘要中参数统一显示为 `?`，不显示敏感值。执行前再次以 PostgreSQL AST 解析器验证为单条 `SELECT` / `WITH ... SELECT`，并拒绝危险函数、系统 Schema 与多语句。

### 5.2 语义校验顺序

1. 校验调用者具备数据源与查询功能权限。
2. 解析同义词，记录每个原词、归一词、命中规则和置信度。
3. 解析概念并校验概念处于已发布版本、归属当前数据源且调用者有权使用。
4. 展开定义、默认时间范围、固定排除条件和行级策略谓词。
5. 转换枚举值并对时间、数值、集合操作符执行类型校验。
6. 校验关联图、聚合粒度、排序字段、明细/聚合兼容性、复杂度、限制行数。
7. 编译并做 SQL AST 白名单验证，再在只读事务中执行。

## 6. 意图、表达与模型边界

意图层仅允许 `QUERY`、`CLARIFY`、`OUT_OF_SCOPE` 三种结果。受限表达层包含原始时间词、候选概念、筛选词、比较关系、分组与排序诉求，不携带 SQL。

规则解析优先覆盖明确的字段/枚举/日期表达；模型只在规则不能完整产生表达时补充结构化 JSON。模型输入使用候选语义资产的短摘要和 JSON Schema，禁止提供连接信息、完整 DDL、未授权表列、敏感枚举或历史结果。模型输出必须经过 Schema 校验、同义词/概念二次解析与权限校验；任何不通过的输出均降级为澄清，不做 SQL 修复重试。

结果总结模型只接收经过列脱敏、行数裁剪和统计聚合后的 `ResultSummary`，不能再次请求数据源。总结必须能链接到使用的指标定义、过滤条件、时间范围和数据更新时间；无结果时解释“按何口径未命中”，不编造结论。

## 7. 持久化模型

建议由新的 Flyway 迁移创建以下平台库表，所有表均包含项目归属、创建/更新时间和必要的操作者审计字段：

| 表 | 用途 |
| --- | --- |
| `data_source_connection` | 数据源基本信息、密文凭据/密钥引用、状态、探测范围、连接限额 |
| `data_source_schema_snapshot` | 数据源元数据快照头、版本、状态、差异摘要 |
| `data_source_schema_object` | 快照内表/列/约束/注释等对象 |
| `data_semantic_model` | 项目或全局语义模型逻辑主记录及当前发布版本 |
| `data_semantic_model_version` | 草稿/发布/废弃版本、快照引用、发布说明 |
| `data_semantic_concept` | 概念、类型、定义 JSON、默认约束与启停状态 |
| `data_semantic_term` | 规范词、同义词、作用域、优先级和冲突状态 |
| `data_semantic_enum_dictionary` | 枚举字典头与版本 |
| `data_semantic_enum_item` | 显示值、别名、物理值、有效期与排序 |
| `data_query_policy` | 数据源/概念授权、行级谓词模板、列脱敏、限额 |
| `data_query_request` | 原问题、解析表达、DSL、版本快照、状态、错误/澄清原因 |
| `data_query_execution` | SQL 摘要、参数脱敏摘要、耗时、行数、结果摘要、预算与追踪标识 |

对具有修改历史和发布语义的资产，禁止运行期原地更新已发布记录；新版本复制后编辑，发布动作通过事务原子切换指针。枚举物理值一旦被发布版本引用，不允许无审计删除，只允许停用或新版本替换。

## 8. 权限、审计与数据保护

新增功能权限：`data-workbench:source-manage`、`data-workbench:semantic-manage`、`data-workbench:query`、`data-workbench:query-audit`。现有 `data-workbench:config` 不自动包含新权限，避免把实体配置管理员提升为外部库访问者。

每个数据源还需有项目可见性和数据源访问授权；每个概念可配置角色/项目范围。行级策略以受控模板表达，如 `tenant_id = :currentTenantId`，模板只允许预定义上下文参数，编译器强制与用户筛选使用 `AND` 合并，不能被 DSL 覆盖。敏感列默认拒绝；获授权后可返回掩码、哈希、区间或聚合值，且摘要模型只能看到脱敏结果。

审计记录至少保存：用户、项目、数据源、请求原文、归一链路、语义/快照/策略版本、DSL、SQL 摘要、脱敏参数、执行耗时/行数、结果摘要、拒绝或澄清原因。原始结果默认不长期落库；如确有导出需求，应走对象存储、数据分级、过期清理和单独授权。

## 9. 接口与前端交互

项目级接口统一位于 `/api/data-workbench/projects/{projectId}`：

```text
POST /data-query/interpret     # 原文 -> 意图、表达、归一链路、候选/澄清
POST /data-query/preview       # 表达 -> 已校验 DSL 与 SQL 摘要，不执行
POST /data-query/execute       # 已确认 DSL -> 执行结果与总结
GET  /data-query/requests/{id} # 查询过程、版本、审计摘要
GET  /data-query/history       # 当前用户查询历史
```

管理端新增「数据源」「语义模型」「词汇与同义词」「字典枚举」「查询策略」「查询审计」六个入口。语义模型编辑器应同时展示业务视图（概念/定义/示例）和只读物理映射（快照表列）；发布前进行冲突、引用、权限、枚举、关联图和示例 DSL 校验。用户查询界面应逐步展示“我理解为”“使用口径”“过滤条件”“数据更新时间”，并对低置信度和高风险明细查询要求确认。

## 10. 与现有 DataChange 的关系

DataQuery 与 DataChange 共用 `DataWorkbenchPermissionSupport`、项目可见性、操作日志和审计体验，但采用独立 DSL：`SemanticQueryDsl` 与 `DataChangeDsl` 不可互转。DataChange 仍由 `DataChangeSqlExecutor` 依据实体字段白名单生成 `UPDATE`，不允许外部数据源和自然语言 SQL。

后续若要支持“查询后发起变更”，必须由用户从查询结果显式选择实体和主键，再创建现有 DataChange 工单；不得将 DataQuery 的 SQL 或结果直接转换为写 SQL。这一隔离是防止读取链路绕过审批、回滚与影响行阈值的关键边界。

## 11. 分阶段实施与验收

### 阶段一：安全只读骨架

- 实现 PostgreSQL 数据源登记、密文凭据、测试连接、Schema 快照与失效检测。
- 实现语义模型版本、概念定义、同义词、枚举和发布校验。
- 支持规则解析、单数据源单实体的聚合/筛选/分组、DSL 预览、参数化执行和审计。
- 验收：无权限、过期快照、非只读连接、无时间范围明细查询、SQL 注入输入和超时查询均被拒绝或安全终止。

### 阶段二：可解释自然语言与治理

- 接入受限 JSON 模型表达，支持澄清卡、定义引用、结果总结和用户历史。
- 加入行级策略、列脱敏、配额、导出审批和管理端审计检索。
- 验收：同义词/定义/字典的发布后变更可追溯；模型输出非法字段或 SQL 片段时不会触达数据源。

### 阶段三：扩展性

- 支持经评审的多表关联模板、更多时间语义、定时刷新与数据质量提示。
- 通过 `SqlDialect` / `MetadataScanner` 插件支持其他数据库；每种方言须独立通过 SQL AST、安全事务和回归样例验证。
- 验收：新增数据库方言不修改语义 API、审计模型或权限模型，并具备独立的连接、超时和注入测试。

## 12. 验证基线

- 后端单元测试覆盖同义词冲突、概念歧义、定义版本冻结、枚举别名、策略强制追加和 DSL 类型校验。
- PostgreSQL 集成测试覆盖只读账户、参数化绑定、`statement_timeout`、`LIMIT`、多语句/危险函数拒绝与快照过期。
- 接口测试覆盖项目隔离、数据源授权、敏感列脱敏、审计完整性、模型 JSON 校验失败降级。
- 前端构建验证管理端配置页与公众端查询页；发布前以固定业务样例回归“意图 → 表达 → 归一 → 概念 → 定义 → 枚举 → DSL → SQL → 总结”全链路。
