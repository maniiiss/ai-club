# 技术债追踪表

## 文档定位

本文档用于记录和追踪项目中的技术债务、遗留问题和待优化项，确保技术债可视化、可管理、可追踪。

适用场景：

- 评审新需求时，评估是否需要先偿还技术债
- 版本规划时，安排技术债整改计划
- 代码审查时，识别新增技术债风险
- 定期回顾，推动技术债持续清理

---

## 技术债分类

### 严重程度

| 等级 | 标识 | 说明 | 处理优先级 |
|------|------|------|-----------|
| P0 | 🔴 | 阻塞性问题，影响核心功能或安全 | 立即处理 |
| P1 | 🟡 | 高风险问题，影响系统稳定性 | 当前迭代 |
| P2 | 🟢 | 中等问题，影响开发效率或用户体验 | 下个迭代 |
| P3 | ⚪ | 低优先级，改善型优化 | 有空处理 |

### 类型分类

| 类型 | 说明 |
|------|------|
| 代码质量 | 代码规范、可读性、可维护性问题 |
| 架构设计 | 架构不合理、扩展性差、耦合过重 |
| 技术选型 | 过时依赖、不合适的框架选择 |
| 测试覆盖 | 缺少测试、测试不充分、测试维护成本高 |
| 文档缺失 | 文档过时、缺少关键文档 |
| 性能问题 | 响应慢、资源占用高、扩展瓶颈 |
| 安全风险 | 潜在安全漏洞、权限问题 |

## 活跃技术债

### TD-001: 前端样式文件过大

| 属性 | 内容 |
|------|------|
| **标识** | TD-001 |
| **标题** | `styles/index.css` 单文件过大（3000+ 行） |
| **类型** | 代码质量 |
| **严重程度** | P2 🟢 |
| **发现日期** | 2026-05-31 |
| **负责人** | 待分配 |
| **状态** | 待处理 |

**问题描述**：

全局样式文件 `frontend/src/styles/index.css` 已超过 3000 行，包含多种组件样式覆盖和响应式规则，难以维护和定位问题。

**影响范围**：

- 样式修改容易产生意外影响
- 新人上手成本高
- 样式冲突排查困难

**建议方案**：

1. 按组件类型拆分样式文件
2. 提取通用工具类
3. 使用 CSS Modules 或 Scoped 样式隔离

**整改计划**：

- [ ] 分析样式依赖关系
- [ ] 制定拆分方案
- [ ] 分阶段重构，保持向后兼容

### TD-002: 部分页面缺少移动端适配

| 属性 | 内容 |
|------|------|
| **标识** | TD-002 |
| **标题** | 自升级中心等页面未接入移动端瀑布流 |
| **类型** | 用户体验 |
| **严重程度** | P2 🟢 |
| **发现日期** | 2026-05-31 |
| **负责人** | 待分配 |
| **状态** | 待处理 |

**问题描述**：

自升级中心等页面尚未按照移动端设计方案进行适配，未接入瀑布流分页能力。

**影响范围**：

- 移动端用户体验不一致
- 部分功能移动端不可用

**建议方案**：

1. 参照 `mobile-console-technical-design-v1.md` 进行适配
2. 使用 `useMobileWaterfallPagination` 组合式工具
3. 统一列表页移动端交互模式

**整改计划**：

- [ ] 梳理未适配页面清单
- [ ] 按优先级排期适配
- [ ] 新增页面强制要求移动端适配

### TD-003: API 层缺少统一错误处理

| 属性 | 内容 |
|------|------|
| **标识** | TD-003 |
| **标题** | API 错误处理分散，缺少统一拦截机制 |
| **类型** | 代码质量 |
| **严重程度** | P2 🟢 |
| **发现日期** | 2026-05-31 |
| **负责人** | 待分配 |
| **状态** | 待处理 |

**问题描述**：

各 API 模块的错误处理逻辑分散，部分接口缺少统一的错误拦截和用户提示机制。

**影响范围**：

- 错误处理代码重复
- 用户体验不一致
- 错误日志收集困难

**建议方案**：

1. 在 `http.ts` 中统一添加响应拦截器
2. 定义标准错误码处理流程
3. 统一用户提示组件

**整改计划**：

- [ ] 设计统一错误处理方案
- [ ] 重构 http.ts 拦截器
- [ ] 逐步迁移各 API 模块

### TD-004: 组件缺少单元测试

| 属性 | 内容 |
|------|------|
| **标识** | TD-004 |
| **标题** | 前端组件单元测试覆盖率低 |
| **类型** | 测试覆盖 |
| **严重程度** | P2 🟢 |
| **发现日期** | 2026-05-31 |
| **负责人** | 待分配 |
| **状态** | 待处理 |

**问题描述**：

前端组件目前缺少单元测试，主要依赖人工测试，回归测试成本高。

**影响范围**：

- 重构风险高
- 回归测试依赖人工

2. 优先覆盖核心业务组件
3. 建立测试覆盖率门禁

**整改计划**：

- [ ] 搭建测试框架
- [ ] 编写核心组件测试
- [ ] 逐步提升覆盖率

## 使用指南

### 新增技术债

1. 在"活跃技术债"章节添加新条目
2. 填写完整信息：标识、标题、类型、严重程度等
3. 明确问题描述、影响范围和建议方案
4. 制定整改计划和负责人

### 状态流转

```
待处理 → 进行中 → 已完成
    ↓
  已取消（如果决定不修复）
```

### 定期回顾

- **每周**: 检查 P0/P1 级别技术债进展
- **每迭代**: 评估技术债整改计划执行情况
- **每月**: 更新统计概览，调整优先级

### 与需求关联

在需求文档或用户故事中引用技术债编号：

```
前置条件：需先完成 TD-001（样式文件拆分）
```
