## 背景

当前 Hermes 已经接入项目、工作项、执行任务、测试计划、Wiki、附件理解等平台内能力，但用户在抽屉界面里主要只能看到“你可以这样问”“需要你确认”“可执行动作”等局部提示，缺少对整体能力边界和确认机制的清晰解释，容易形成两类误解：

- 误以为 Hermes 是任意外部 agent，应该直接联网、打开浏览器或绕过平台数据边界。
- 误以为 Hermes “只会建议不会执行”，无法理解候选卡片和确认动作其实是平台有意加入的安全机制。

第一版先不改后端 Prompt、MCP 工具注册和接口协议，只通过前端悬浮提示与最小状态文案增强讲清楚“平台内协作助手”的定位与候选/动作确认流程。

## 目标

- 在 Hermes 标题右侧增加 `?` 悬浮提示，集中承载能力边界与确认机制说明。
- 统一前端对 Hermes 的口径：
  - Hermes 是平台内协作助手
  - 查询会优先结合当前页面上下文和平台内数据
  - 写操作会先生成待确认动作
  - 歧义对象会先要求用户选择候选
- 把流式中断提示改成更贴近当前机制的说明，避免用户误认为 Hermes 整体不可用。

## 范围

本方案只覆盖前端展示层与文案，不包含：

- 后端 Prompt 资源变更
- Tool Registry、MCP Tool、会话状态结构变更
- 新增结构化能力元数据接口
- Hermes 新能力接入

## 风险与依赖

| 风险/依赖 | 当前状态 | 应对 |

## 当前结论

第一版已经以最低成本把 Hermes 的产品定位和确认机制前移到前端展示层：用户现在可以通过标题右侧 `?` 悬浮提示理解“这是平台内协作助手，它为什么会停下来等你确认”，同时正文区保持相对简洁。后续如仍需继续增强，再评估结构化能力面板、前端显式 Skill 标识或后端能力元数据接口。
