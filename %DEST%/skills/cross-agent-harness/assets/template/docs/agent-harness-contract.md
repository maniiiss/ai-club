# 跨智能体 Harness 协议

## 目的

本文件定义仓库内通用的智能体协作协议。任何能够读取仓库文件的智能体，都应优先依据本文件、`AGENTS.md` 和 `docs/` 内正式文档开展工作，而不是依赖平台专有技能系统或历史聊天上下文。

## 适用范围

本协议适用于：

- 使用仓库开展分析、设计、实现、校验和文档维护的智能体。
- 需要在不同智能体产品之间共享工作方式、计划和验证规则的场景。

本协议不覆盖用户的直接指令；当用户明确提出要求时，以用户指令为最高优先级。

## 信息源优先级

智能体默认按以下顺序判断信息优先级：

1. 用户直接指令。
2. `AGENTS.md` 中的仓库入口与硬约束。
3. `docs/` 下的正式文档、架构文档和设计文档。
4. 代码、脚本、测试及其注释。
5. 历史聊天上下文。

这条规则的业务意图是避免智能体持续沿用已经过期的聊天结论，而忽略仓库中已经更新的正式事实源。

## 标准工作流程

智能体在开始修改前，默认遵循以下顺序：

1. 阅读 `AGENTS.md`，定位入口和硬约束。
2. 阅读与当前任务相关的协议、架构、设计或执行计划文档。
3. 阅读相关代码、脚本和测试，确认当前实现边界。
4. 选择最小可验证 harness。
5. 执行修改。
6. 在需要时补充或更新设计文档、执行计划和索引。
7. 运行验证。
8. 在交付说明中记录已验证范围和未验证风险。

## 文档落盘规则

### 入口文档

`AGENTS.md` 只承担目录和高频规则入口职责，不承载长篇设计、详细接口表和历史排障记录。

### 正式事实源

以下内容必须优先落到 `docs/`：

- 系统边界和核心链路。
- 技术架构调整。
- 跨模块协议。
- 重要实现设计。
- 执行计划。
- 验证方式和已知风险。

### 计划文档

执行计划应放在 `docs/exec-plans/active/`，完成后移动到 `docs/exec-plans/completed/`。计划至少包含：

- 目标
- 范围
- 非目标
- 影响模块
- 前置条件
- 实施步骤
- 验证方式
- 风险与回滚
- 当前状态
- 遗留事项

### 设计文档

涉及技术架构调整、重要模块边界变化、公共协议设计或复杂跨模块变更时，必须更新 `docs/architecture.md` 或新增正式设计文档。

### 索引同步

新增正式设计文档后，必须同步更新 `docs/design-docs/index.md`，确保后续智能体能发现这些文档。

## 验证规则

### 最小验证

任何修改都应优先选择与当前改动最相关、反馈最快的验证动作。

### 扩展验证

在重大协议调整、目录变化或脚本变更后，应补充更完整的兜底验证，例如运行自动化测试或人工检查关键入口是否可发现。

### 文档与脚本类变更

文档、目录和脚本改动至少应执行：

- `python scripts/validate_harness.py`
- `python -m pytest tests/test_validate_harness.py -q`

若环境不满足执行条件，应明确说明未验证原因和潜在风险。

## 交付物规则

一次合格交付至少应留下以下资产中的适用部分：

- 代码或文档改动。
- 必要的设计文档或执行计划。
- 验证命令与结果。
- 剩余风险说明。

如果改动涉及公共协议、结构或架构边界，只改代码但不补文档，不算完成。

## 文档维护规则

- 入口失效后要立即修复，不允许长期依赖聊天补充入口。
- 文档结构变化后，同步更新索引和校验脚本。
- 发现过期说明时，应优先修正文档而不是依赖口头例外。
- 新增约束后，优先考虑是否能被脚本机械检查。

## 当前机械校验范围

第一版 harness 校验聚焦高价值结构项：

- 必需入口文件与目录存在。
- `AGENTS.md` 包含关键文档和校验命令入口。
- 关键文档可按 UTF-8 读取。

更复杂的语义检查和基于 Git diff 的规则将在后续迭代中补充。
