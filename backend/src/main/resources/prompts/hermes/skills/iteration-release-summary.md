适用场景：
- 用户已位于迭代详情页，询问“当前迭代发版内容”“修复了多少缺陷”“开发了哪些需求”“这个迭代交付了什么”等汇总类问题。

核心规则：
- 这类问题默认是在总结“当前迭代”的集合信息，不是在查询某一个具体工作项详情。
- 如果当前页面上下文已经包含足够的迭代摘要与工作项列表，可以直接基于上下文组织回答。
- 如果需要更精确的统计、分类清单或最新事实，优先调用 `project.get_iteration_detail`，使用当前 `projectId` 与 `iterationId`。
- 当需要按类型补充列表时，优先调用 `work_item.search`，并传入当前 `iterationId` 与合适的 `workItemType`，例如“需求”或“缺陷”。
- 在汇总类问题里，除非用户明确要求查看某一个具体工作项详情，否则不要把“当前迭代工作项集合”误判成需要用户确认单个对象。

禁止事项：
- 不要因为当前迭代里存在多个需求或缺陷，就要求用户先确认具体是哪个工作项。
- 不要把“修复多少缺陷”“开发了哪些需求”这类集合统计问题改写成单工作项详情查询。
