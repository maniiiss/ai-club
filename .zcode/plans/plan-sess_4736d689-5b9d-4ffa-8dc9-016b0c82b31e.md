# gitpilot-cli 改造计划:fork pi-coding-agent 0.81.1 二开

## 背景与目标架构

当前 `gitpilot-cli/` 是从零手搓的 CLI(terminal-ui/local-tools/output/session-store/main.ts),重复造了 pi-coding-agent 已自带的能力。改为 **fork `@earendil-works/pi-coding-agent@0.81.1` 源码进仓库**,改品牌为 gitpilot,平台对接全部做成 pi extension,核心仍是 pi。

目标架构:

```
gitpilot-cli/                        # = forked pi-coding-agent 0.81.1 源码,改品牌
  package.json                       # name @aiclub/gitpilot-cli, bin{gitpilot}, piConfig{gitpilot,.gitpilot}
                                      # deps: @earendil-works/pi-agent-core^0.81.1 / pi-ai / pi-tui
  tsconfig.build.json                # 改为 standalone(内联 base,去 monorepo workspace 引用)
  src/                               # coding-agent 源码(改品牌字面量)
    extensions/gitpilot/
      platform-auth.ts               # /login /logout /status + keyring + 平台地址配置(替代 api/config/credentials)
      platform-model.ts              # 自定义 provider + before_provider_request 接平台模型网关(替代 resolvePiModel+modelSession)
      index.ts                       # 汇总注册
  docs/ tests/                       # fork 自带(改品牌)+ 新增 extension 测试

packages/gitpilot-agent-core/        # 收瘦:仅 handoff 协议 + pi-runtime 用 agent factory
  src/index.mjs                      # import 改 @earendil-works/*;保留 createPiAgent/resolvePiModel/normalizePiEvent(toPiHistoryMessages(给 pi-runtime)+ handoff 协议
  cloud/                             # 停车:吸收 gitstate/snapshot.ts + handoff/(后续云端接力 extension 用)

pi-runtime/                          # 升级 @earendil-works 0.81.1,Docker node:20->node:22
backend / frontend-public            # 不改(设备授权/模型代理/scope/V143/设备授权页全部保留,作为 extension 的对接 API)
```

## 阶段 1:平台对接 extension(让 gitpilot login + 平台模型可用)

复用后端已有的 6 个 endpoint(不改后端):

### 1.1 platform-auth extension(`extensions/gitpilot/platform-auth.ts`)
- `registerCommand("/login")`:POST `/api/cli/device/authorizations` 拿 deviceCode/userCode/verificationUri → `open-browser` → 按 intervalSeconds 轮询 `/api/cli/device/token`(428 继续/410 过期)→ 拿 gpt_ token 存 `@napi-rs/keyring`(key=platformUrl)。
- `registerCommand("/logout")`:POST `/api/cli/logout` + 删 keyring。
- `registerCommand("/status")`:GET `/api/cli/me` 显示用户。
- 平台地址:写入 `~/.gitpilot/agent/settings.json` 的自定义字段(替代 config.ts);`registerPlatform` 逻辑并入 /login 首次引导。

### 1.2 platform-model extension(`extensions/gitpilot/platform-model.ts`)
- 启动/会话开始时,用 gpt_ token GET `/api/cli/models` 拿平台模型清单,`registerProvider("gitpilot", {...})` 把每个平台模型注册为自定义 provider 条目(provider=OPENAI→api openai-completions / ANTHROPIC→messages)。
- `on("before_provider_request")`:当目标模型是 gitpilot 平台模型时,确保有效 model session(POST `/api/cli/model-sessions` 拿 proxyBaseUrl+gms_ token,缓存 modelConfigId→{proxyBaseUrl, token, expiresAt},临近 900s 过期则重建),改写请求:URL→`{proxyBaseUrl}/chat/completions` 或 `{proxyBaseUrl}/messages`,header 注入 `Authorization: Bearer {gms_}`(openai)或 `x-api-key: {gms_}`+`anthropic-version`(anthropic)。服务端已强制 stream/model 覆盖。
- 默认模型源设为平台;pi 原生 BYOK/provider 配置仍保留(高级用户可用 `~/.gitpilot/agent/models.json`)。
- 验收:`gitpilot login` → `gitpilot`(交互)能用平台模型跑通 coding agent + 工具调用 + 用量统计;`/models` 切换平台模型;session 过期自动重建。

1. **platform-model extension 的 before_provider_request 改写**是核心实现难点(动态 session 刷新 + provider/header 改写);先用 vitest mock 平台 API 验证,再端到端。

4. **fork 升级维护**:源码进仓库后,后续 pi 升级需手动 merge;接受此权衡(用户已选 fork 而非依赖+扩展)。
