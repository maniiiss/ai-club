# GitLab 推送到业主代码仓库 - 实现方案

## 一、需求与方案概览

新增"手动把平台 GitLab 仓库的代码推送到业主 GitLab 仓库"的交付能力。已确认:
- **目标平台**:GitLab(其他实例)
- **推送内容**:整个分支镜像,保留完整提交历史 -> **必须用原生 `git push`**(GitLab Commits REST API 只能创建新 commit,无法推送历史 commit 链)
- **推送方式**:支持多种可选(直接 push / 推到新分支 / 创建 MR 让业主审批)
- **凭据管理**:按项目独立配置业主仓库地址 + token
- **两端都要实现**:管理端 `frontend`(配置绑定 + 推送 + 历史)和公众端 `frontend-public`(推送 + 历史,沿用"管理端配置、公众端操作"模式)

## 二、关键架构决策

**决策:git push 能力放在 `code-processing` 服务,backend 负责编排。**

理由:
1. `code-processing` 已有完整的 git subprocess 基础设施 -- 3 种认证策略(`oauth2:token@` basic-auth / `PRIVATE-TOKEN` header / `Bearer` header)、URL 鉴权构造、`GIT_TERMINAL_PROMPT=0`、敏感信息脱敏(`_sanitize_sensitive_text`)。新增 push 只是在 clone 之后加 `git push` 命令,认证策略**完全复用**。
2. 符合现有架构分工:backend 做编排/凭据解密/权限/持久化,code-processing 做代码操作。大仓库 clone+push 是长任务,FastAPI async + 长超时(现有 180s)模型比 backend 短请求模型更合适。
3. backend↔code-processing 内部调用鉴权已有 `_require_internal_service_auth`(`Bearer {internal_service_token}`),凭据传递安全有保障。

**架构边界变化(需文档化)**:`code-processing` 从"纯只读分析服务"扩展为"可执行代码推送"。这是合理演进,需在 `docs/design-docs/` 新增专题设计文档,并更新 `docs/architecture.md` 服务边界说明(遵循 AGENTS.md 要求)。

## 四、后端实现(backend, Spring Boot)

### 4.1 实体
- `ProjectOwnerRepoBindingEntity`(@Table=project_owner_repo_binding,字段同表1,带 @PrePersist/@PreUpdate)
- `OwnerRepoPushLogEntity`(@Table=owner_repo_push_log)
- 中文注释说明业务意图(AGENTS.md 要求)

### 4.2 Repository
- `ProjectOwnerRepoBindingRepository`(JpaRepository + JpaSpecificationExecutor):`findByProject_IdOrderByIdAsc`、`existsByProject_IdAndApiBaseUrlAndGitlabProjectRef`
- `OwnerRepoPushLogRepository`:`findByBinding_IdOrderByIdDesc(bindingId, Pageable)`

### 4.3 Service
- **`OwnerRepoPushService`**(参照 `GiteeTestPlanPushService` 编排):
  - `getPushContext(bindingId)` - 返回 `{canPush, disabledReason, lastPushStatus, lastPushedAt, lastPushMessage}`(复用 Gitee push-context 范式)
  - `pushToOwnerRepo(bindingId, {sourceBindingId, sourceBranch, targetBranch, pushMode})` @Transactional:
    1. `projectDataPermissionService.requireProjectVisible` 校验项目权限
    2. 校验业主绑定启用 + 解密业主 token(`tokenCipherService.decrypt`)
    3. 校验源绑定 + 解密源 token
    4. 调 code-processing 镜像推送接口(传源/目标 clone url+token+分支+pushMode)
    5. 若 pushMode=MERGE_REQUEST:调 `GitlabApiService.createMergeRequest`(指向**业主** apiBaseUrl + 业主 token,从临时分支到目标主分支)
    6. 落库 `owner_repo_push_log`(状态/commit sha/MR 信息)
    7. 更新绑定 `last_push_status/message/pushed_at`
    8. 返回三态结果(SUCCESS/PARTIAL/FAILED)
- **`OwnerRepoBindingManagementService`**:绑定 CRUD + test 连接(调 `GitlabApiService.fetchProject` 校验业主 token)+ 拉取业主仓库元信息填充

### 4.4 Controller
`OwnerRepoController`(@RequestMapping("/api/gitlab/owner-repos"),全部 @RequirePermission):
- `GET /bindings`(分页查询,权限 gitlab:view)
- `POST /bindings` / `PUT /bindings/{id}` / `DELETE /bindings/{id}`(权限 gitlab:owner-repo:manage)
- `POST /bindings/{id}/test`(测试业主仓库连接)
- `GET /bindings/{id}/push-context`(推送前置上下文)
- `POST /bindings/{id}/push`(触发推送,payload:sourceBindingId/sourceBranch/targetBranch/pushMode)
- `GET /bindings/{id}/push-logs`(推送历史分页)

> 放在 `/api/gitlab/owner-repos` 下,语义上归属 GitLab 模块,前端复用 gitlab:view 菜单入口。

### 4.5 code-processing 调用客户端
新增 `OwnerRepoPushClientService`(Java HttpClient,调 code-processing 内部接口,带 `Bearer {internal_service_token}`),参照现有 `GitlabCodeStructureClientService` 的调用模式。

## 五、code-processing 实现(FastAPI)

### 5.1 新增镜像推送服务
`code-processing/app/services/owner_repo_push_service.py`:
- 复用 `repository_scan_service.py` 的 3 种认证策略(`_build_authenticated_repo_url` + `http.extraHeader`),提取为共享工具函数
- 核心函数 `mirror_push_to_owner_repo(request)`:
  1. `git clone`(完整历史,**去掉现有 `--depth 1`**)源仓库指定分支到临时工作区(复用 3 策略 + http->https 兜底)
  2. 添加目标仓库为 remote:`git remote add target <带鉴权的目标 url>`
  3. 按 pushMode 执行:
     - `DIRECT`:`git push target <src>:<target> --force-with-lease`(目标分支历史不同需强制,见风险说明)
     - `NEW_BRANCH`:`git push target <src>:refs/heads/delivery/<timestamp>`
     - `MERGE_REQUEST`:`git push target <src>:refs/heads/delivery/<timestamp>`(backend 再调 createMergeRequest)
  4. `git rev-parse HEAD` 取源 commit sha;`git ls-remote target <target>` 取目标 commit sha
  5. 清理临时工作区
  6. 返回 `{sourceCommitSha, targetCommitSha, pushedBranch, strategy}`;失败抛异常带脱敏消息(`_sanitize_sensitive_text`)
- 工作区用临时目录(`tempfile.mkdtemp`),推送完即删(不像代码结构那样持久缓存,因为推送是一次性交付动作)

### 5.2 新增路由
`code-processing/app/routers/owner_repo_push.py`(参照现有 router 风格):
- `POST /owner-repo-push/mirror`(@require_internal_service_auth)-- 接收 `OwnerRepoMirrorPushRequest`(源/目标 repoUrl+authToken+apiBaseUrl+projectRef、sourceBranch、targetBranch、pushMode),调 `mirror_push_to_owner_repo`
- 在 `app/main.py` 注册 router

### 5.3 请求模型
`code-processing/app/models.py` 新增 `OwnerRepoMirrorPushRequest`(字段同上,Pydantic),参照现有 `RepositoryScanPrepareRequest`。

## 八、推送流程时序(以 MR 方式为例)

```
前端(管理端/公众端) POST /api/gitlab/owner-repos/bindings/{id}/push
  └─ backend OwnerRepoPushService.pushToOwnerRepo
       1. 校验权限 + 解密业主token + 解密源token
       2. POST code-processing /owner-repo-push/mirror  (Bearer internal_service_token)
            └─ code-processing: git clone 源(完整历史) -> git remote add target -> git push target src:delivery/xxx
            └─ 返回 {sourceCommitSha, pushedBranch}
       3. pushMode=MERGE_REQUEST -> GitlabApiService.createMergeRequest(业主apiBaseUrl, 业主token, delivery/xxx -> 目标主分支)
       4. 落库 owner_repo_push_log + 更新 binding.last_push_*
       5. 返回三态结果
```

## 九、风险与注意事项

1. **force push 覆盖风险(最重要)**:`DIRECT` 模式下,若业主仓库目标分支已有不同历史,需 `--force-with-lease` 强制覆盖。前端推送确认弹窗必须强提示"将覆盖业主仓库 {targetBranch} 分支的历史,不可恢复",并要求二次确认。推荐默认 pushMode 设为 `NEW_BRANCH` 或 `MERGE_REQUEST`(更安全)。
2. **长任务超时**:大仓库 clone+push 可能数分钟。code-processing 接口超时设 600s;backend 调 code-processing 的 HttpClient 超时同步放大;前端推送接口单独用更长 timeout(管理端 http.ts 全局 60s,推送请求需覆盖 timeout;公众端同理)。**第一版建议:同步执行 + 长超时 + 全屏 loading;后续可演进为异步任务**(参照仓库扫描的异步任务模式)。
3. **token 传递安全**:backend->code-processing 传明文 token(内部网络 + Bearer 鉴权);code-processing 日志/异常必须经 `_sanitize_sensitive_text` 脱敏(token 和 `quote(token)` 都替换为 `***`)。
4. **目标分支不存在**:GitLab 直接 push 不存在的分支会自动创建,无需预处理。
5. **空目标仓库**:首次 push 无问题。
6. **公众端 Select 强制**:`uiHarness.test.ts` 校验全仓不得出现原生 `<select>/<option>`,新增表单必须用 `src/components/common/Select.tsx`。

## 十三、第一版明确不做(控制范围)

- 定时/自动推送(用户明确要手动)
- 远端分支/仓库清理
- 推送到非 GitLab 平台(用户已确认目标就是 GitLab;若未来要支持 GitHub/Gitee,再抽象平台层)
