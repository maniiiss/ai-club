# 工作项驱动的 PRD 自动沉淀实施计划

## 1. 背景与目标

当前希望把“需求录入”能力沉淀为一套可持续演进的 PRD 机制，但核心前提已经明确：

- **主数据来源是工作项需求录入**
- **Hermes 只是辅助入口，不是主数据源**
- **Hindsight 负责召回相关上下文，不直接生成 PRD**
- **真正的完善动作由大模型完成，最终结果需要用户确认后再回写**

本计划的目标是：在不改变现有工作项作为业务主入口的前提下，为已绑定 Wiki 空间的项目自动建立 PRD 承载结构，并逐步形成“工作项主数据 + PRD 文档镜像 + Hindsight 检索增强 + LLM 辅助完善”的闭环。

---

## 2. 核心原则

### 2.1 主从关系

#### 主数据
- 工作项基础字段
- 工作项描述
- `requirementMarkdown`
- 工作项关联信息（项目、模块、迭代、状态、负责人等）

#### 派生数据
- Wiki 空间中的 PRD 页面或目录
- PRD 章节化内容
- 需求变更记录、待确认问题等文档内容

#### 辅助能力
- Hermes：交互入口、补全建议、追问和整理
- Hindsight：召回相关历史需求、PRD、会议纪要、上下文
- LLM：基于当前需求与召回结果生成补全建议、缺口分析和合并草稿

### 2.2 事实来源约束

- 工作项永远是业务主键
- PRD 文档是工作项需求的知识化投影，不作为独立主事实源
- Hermes 生成的内容默认先作为“建议”，不直接写入正式事实
- Hindsight 保存的是可召回知识，不替代业务表

---

## 3. 目录与承载模型

## 3.1 推荐层级

建议采用：

- **项目 = 空间**
- **模块 = 分类目录**
- **需求 = 默认一篇 PRD 页面**

推荐目录结构：

```text
项目空间
└── 需求PRD
    └── 模块目录
        ├── REQ-101-自动生成PRD
        ├── REQ-102-需求补全建议
        └── REQ-103-历史需求召回
```

### 3.2 为什么默认“一需求一页面”

第一阶段不建议直接做“一需求一整套目录”，原因如下：

- 需求主实体是工作项，不是目录
- 页面比目录更轻，便于快速跑通自动创建与同步
- 单页已经能承载绝大多数 PRD 初稿
- 后续复杂需求仍可升级为“目录 + 多页面”

### 3.3 复杂需求升级策略

当需求满足以下条件之一时，可升级为“一需求一目录”：

- 内容明显超过单页可维护范围
- 需要拆分业务流程、交互说明、接口设计、验收标准
- 需要多人长期协同维护
- 需要独立的变更记录与待确认问题页

## 4. 核心链路设计

## 4.1 基础自动创建链路

触发条件：

1. 新建“需求”类工作项
2. 工作项所属项目已绑定 Wiki 空间
3. 当前工作项尚未绑定 PRD 页面

执行步骤：

1. 读取工作项所属项目、模块、需求内容
2. 查找项目绑定空间
3. 检查空间下是否存在 `需求PRD` 根目录，不存在则创建
4. 根据模块信息查找或创建模块目录
5. 创建该需求对应的 PRD 页面
6. 将工作项当前内容渲染为 PRD 初稿
7. 建立工作项与 PRD 页面映射关系

## 4.2 持续更新链路

当工作项需求内容被修改时：

1. 读取工作项当前最新内容
2. 读取已有 PRD 页面内容
3. 生成新的 PRD 建议稿或增量章节
4. 用户确认后回写 PRD 页面
5. PRD 页面更新后进入 Hindsight 同步链路

## 4.3 AI 完善链路

当用户主动点击“AI 完善 PRD”或在 Hermes 中发起补全诉求时：

1. 读取工作项主数据
2. 读取关联 PRD 当前内容
3. 从 Hindsight 召回当前项目/模块/历史相似需求上下文
4. 调用大模型做缺口分析、章节归类、补全文案生成
5. 生成“建议更新卡片”
6. 用户确认后再回写工作项或 PRD

---

## 5. Hindsight 与大模型的职责边界

## 5.1 Hindsight 的职责

Hindsight 在本方案中只负责“找”，不负责“写”。

主要职责：

- 召回当前项目历史相似需求
- 召回当前模块历史 PRD
- 召回当前需求相关会议纪要、补充说明、变更记录
- 通过标签和范围控制为 LLM 提供最相关的上下文

不负责：

- 自动生成 PRD
- 判断最终内容是否正确
- 直接写回业务数据

## 5.2 大模型的职责

大模型负责真正的“完善”动作，包括：

- 识别当前输入应该归入哪个 PRD 章节
- 识别当前 PRD 缺失哪些信息
- 基于召回内容产出补全建议
- 生成待确认问题
- 生成合并后的 PRD 草稿

## 5.3 切入点定义

大模型不应只绑定在 Hermes 聊天入口，更合理的切入点是：

### 主切入点
- 工作项服务层的“PRD 分析 / PRD 刷新 / 缺口检查”能力

### 辅助切入点
- Hermes 会话中的“帮我补全 PRD”“看看还缺什么”“基于历史需求生成建议”

推荐新增后端能力，例如：

- `task.prd.initialize`
- `task.prd.refresh`
- `task.prd.gap-check`
- `task.prd.suggest-update`

这些能力内部统一执行：

1. 读取工作项
2. 读取 PRD
3. Hindsight Recall
4. LLM 推理
5. 返回建议结果

---

## 6. 数据模型建议

## 6.1 建议新增关系字段

当前系统需要补充工作项与 PRD 之间的稳定映射关系，建议至少支持：

- `taskId`
- `projectId`
- `wikiSpaceId`
- `prdWikiPageId`
- 可选：`prdWikiDirectoryId`

### 建议目标

- 一个需求工作项默认只绑定一个主 PRD 页面
- 后续如升级为目录模式，可在不破坏映射关系的情况下扩展

## 6.2 PRD 页面元数据建议

建议在 PRD 页面内容或元数据中保留：

- 关联工作项 ID
- 模块名称/模块 ID
- 当前同步状态
- 最近一次 AI 建议生成时间
- 最近一次用户确认时间

---

## 7. PRD 初稿生成策略

## 7.1 初稿来源

初稿优先基于工作项当前字段生成：

- 工作项标题
- 工作项描述
- `requirementMarkdown`
- 原型链接（如有）
- 验收相关字段（如有）

## 7.2 初稿结构建议

第一阶段建议采用轻量固定模板：

```markdown
## 背景

## 目标

## 用户故事

## 需求描述

## 范围与边界

## 验收标准

## 待确认问题
```

说明：

- 初稿以工作项已有内容优先填充
- 缺失章节允许保留空段落，但需要标记“待完善”
- 后续 AI 只做增量补全，不强制每次覆盖整篇文档

---

## 8. 同步策略建议

## 8.1 工作项到 PRD

这是主同步方向。

触发方式：

- 新建工作项后初始化
- 编辑需求后手动刷新
- 编辑需求后自动生成“可同步建议”

推荐第一阶段使用：

- **保存后不直接覆盖 PRD**
- **而是生成待确认更新建议**

这样可以避免工作项频繁编辑导致 PRD 抖动。

## 8.2 PRD 到工作项

默认不建议反向自动覆盖主数据。

如果用户在 Hermes 或 PRD 编辑过程中产生了新的结构化建议，应采用：

1. 先生成建议
2. 用户确认
3. 再选择写回工作项、PRD，或同时写回

---

## 9. Hindsight 召回范围设计

建议按优先级逐层召回：

1. 当前需求自身历史 PRD / 变更记录
2. 当前模块下历史相似需求
3. 当前项目空间下相似 PRD
4. 同项目其他知识材料（会议纪要、方案页）

建议标签维度：

- `task:{taskId}`
- `project:{projectId}`
- `module:{moduleKey}`
- `space:{wikiSpaceId}`
- `type:prd`
- `type:meeting-note`
- `type:requirement-change`

召回结果应限制数量与长度，避免 LLM 输入过载。

## 10. 分阶段实施计划

## Phase 1：建立工作项到 PRD 的基础闭环

目标：先把“自动创建 + 手动打开 + 基础同步”跑通。

实施内容：

1. 需求工作项自动检测项目绑定空间
2. 自动创建 `需求PRD` 根目录与模块目录
3. 自动创建需求 PRD 页面
4. 基于工作项内容生成 PRD 初稿
5. 保存工作项与 PRD 页面映射
6. 在工作项详情中增加“打开 PRD”入口

交付结果：

- 工作项有稳定 PRD 落点
- 需求沉淀正式进入空间体系

## Phase 2：引入 AI 分析与建议更新

目标：先让 AI 给建议，但不直接改正式内容。

实施内容：

1. 新增后端 PRD 分析能力
2. 接入 Hindsight 召回
3. 接入大模型补全与缺口分析
4. 生成“建议更新卡片”
5. 用户确认后再回写 PRD

交付结果：

- 支持 PRD 缺口分析
- 支持 AI 建议更新
- 支持基于历史相似需求的补全

## Phase 3：Hermes 辅助协作

目标：让 Hermes 成为 PRD 完善入口之一，但不改变工作项主数据地位。

实施内容：

1. Hermes 会话自动带入 `taskId + wikiSpaceId + prdWikiPageId`
2. 提供“帮我补全 PRD”“看看还缺什么”等提示能力
3. Hermes 内部调用统一 PRD 分析服务
4. 产出建议卡片并支持确认回写

交付结果：

- Hermes 成为自然的需求讨论和 PRD 完善入口
- 但底层仍是工作项驱动

## Phase 4：复杂需求升级与沉淀增强

目标：支持复杂需求的多页面结构与变更沉淀。

实施内容：

1. 支持“一需求一目录”升级
2. 支持待确认问题页、变更记录页
3. 支持章节级别的增量更新
4. 支持按需求阶段切换不同补全策略

---

## 11. 风险与控制点

## 11.1 风险

- 工作项与 PRD 双写导致事实不一致
- AI 生成内容未经确认进入正式文档
- Hindsight 召回范围过大导致提示词污染
- 自动创建目录或页面时出现重复数据
- 复杂需求过早采用多层结构，增加维护成本

## 11.2 控制点

- 明确工作项为主事实源
- AI 输出默认只作为建议
- 目录/页面创建必须幂等
- 召回范围按“当前需求 > 当前模块 > 当前项目”逐级控制
- 默认单需求单页，复杂场景再升级

---

## 12. 当前推荐的第一落点

如果近期只做一件最值的事，建议优先落地：

### 第一优先级
- **工作项创建后自动生成 PRD 页面并建立映射**

### 第二优先级
- **增加“AI 检查 PRD 缺口 / 生成补全建议”动作**

### 第三优先级
- **让 Hermes 调用同一套 PRD 分析能力**

---

## 13. 一句话总结

本方案最终要实现的不是“Wiki 自动建目录”本身，而是：

> **以工作项需求录入为主数据源，把项目空间作为 PRD 承载容器，利用 Hindsight 提供相关上下文，再由大模型在工作项服务层和 Hermes 辅助入口中产出可确认的 PRD 完善建议，最终逐步沉淀出高质量需求文档。**
