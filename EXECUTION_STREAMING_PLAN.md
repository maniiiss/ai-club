# 执行中心 CLI 长任务与流式输出改造方案

## 1. 背景

当前执行中心已经具备“任务 -> 规划 -> 开发 -> 测试 -> 报告”的闭环能力，也已经接入了 Claude Code、Codex 以及 HTTP API / AGENT_RUNTIME 两类真实执行入口。

但在真实使用中，CLI 类步骤逐渐暴露出三个结构性问题：

- **长任务被同步 HTTP 超时约束住**
- **CLI 输出全部一次性展示，页面容易卡顿**
- **执行过程缺少响应式、渐进式的实时反馈**

这些问题在开发执行、测试执行、仓库扫描等长时步骤中会越来越明显，因此需要把当前“同步接口调用模型”升级为“异步任务执行 + 流式事件输出”模型。

---

## 2. 当前问题总结

### 2.1 固定 300s 上限不适合 CLI 长任务

当前实现里，超时实际上是被多层共同限制的：

- Agent 配置中的 `timeoutSeconds`
- backend 对 Agent timeout 的上限裁剪
- code-processing 桥接请求模型里的 `timeoutSeconds <= 300`
- CLI 子进程执行时直接复用该超时时间

这意味着当前 `PLAN`、`IMPLEMENT`、`TEST` 这类本质上可能需要 10 分钟、30 分钟甚至更久的步骤，被压缩成了一次同步 HTTP 请求中的 300 秒窗口。

对短任务这没有问题，但对真实 CLI 扫描、依赖安装、构建、测试、代码修改来说，**300 秒只是“请求等待上限”，不是合理的“步骤执行上限”**。

### 2.2 全量 stdout/stderr 直接展示会拖垮页面

当前执行详情页更偏向于“展示完整步骤快照”：

- `inputSnapshot`
- `outputSnapshot`
- artifact 原文内容

对于 Markdown、短 JSON、简短结论，这种方式很好。

但对于下面这类输出，直接整块渲染会越来越重：

- `pnpm install` 的海量安装日志
- `vite build` 的长告警输出
- `mvn test` 的完整测试日志
- `claude/codex` CLI 的原始过程输出

如果这些内容都以一次性大文本写入页面，问题会越来越明显：

- 页面初次加载慢
- 折叠面板展开卡顿
- 浏览器内存占用持续升高
- 用户难以从大段文本中找到当前最有用的信息

### 2.3 当前进度反馈太粗，不足以支撑“盯执行”

当前详情页主要依赖：

- `progressPercent`
- `latestMessage`
- 步骤结束后的 artifact / output

这能表达“正在跑哪个步骤”，但表达不了更细粒度的信息：

- 当前具体在执行哪个命令
- 最近 10 秒输出了什么
- 是在下载依赖、构建还是打包
- 是真的还在跑，还是已经卡住

也就是说，**用户只能看到步骤级状态，看不到命令级和日志级状态**。

---

## 3. 改造目标

本轮方案希望达成以下目标：

- 支持 CLI 长任务脱离同步 HTTP 300 秒限制
- 支持步骤执行过程中的实时、渐进式反馈
- 支持前端按需展示日志，而不是默认渲染全部原文
- 保持现有执行中心任务、步骤、产物模型继续可用
- 尽量兼容现有 HTTP_API / AGENT_RUNTIME 接入方式，避免一次性推翻重做

---

## 4. 非目标

本轮不追求一次性解决所有问题，以下内容可以后续演进：

- 不要求第一阶段就把所有 Agent 都改成流式
- 不要求第一阶段就支持多路并发命令级细粒度取消
- 不要求第一阶段就引入复杂的 WebSocket 双向协议
- 不要求第一阶段彻底移除现有 artifact 机制

---

## 5. 设计原则

### 5.1 长任务执行必须异步化

CLI 步骤不能再被建模为“后端发起一次 HTTP 请求并阻塞等待完整结果”，而应建模为：

1. 提交步骤执行请求
2. 获得一个执行会话
3. 异步持续产生日志、状态、进度事件
4. 最终汇总结果并落产物

### 5.2 页面默认展示摘要，不默认展示全部日志

用户最先需要看到的不是完整 stdout，而是：

- 当前步骤
- 当前命令
- 最近输出
- 关键结论
- 错误摘要

完整原始日志应成为“按需展开或下载”的能力，而不是默认全量渲染。

### 5.3 前端只和 backend 通信

即使内部存在 `code-processing`、Hermes、其他 runtime，前端也应只订阅 backend 提供的统一事件流。

这样有三个好处：

- 权限与鉴权统一
- 重连与状态恢复统一
- 避免前端直接感知多个内部服务的协议差异

### 5.4 结构化结果和原始日志分开保存

步骤最终结果仍然需要结构化沉淀：

- 执行状态
- 关键摘要
- 结构化 JSON 结果
- Markdown 报告

但日志流应单独建模，不再强行塞进 `outputSnapshot` 或单个 artifact 文本中。

---

## 6. 总体方案

## 6.1 从“同步请求”升级为“异步步骤会话”

建议为长时步骤引入 **Step Execution Session** 概念。

执行链路变为：

1. backend 调度到某个步骤
2. backend 为该步骤创建 `execution session`
3. backend 调用对应执行器时，不再等待最终完整输出，而是先拿到：
   - `sessionId`
   - `acceptedAt`
   - 可选 `runnerPid / workspace`
4. 执行器在后台持续运行 CLI，并不断上报事件
5. backend 持续接收事件并更新步骤状态
6. 步骤结束时，执行器提交最终结构化结果

对于短步骤，仍可以保留同步模式；对于长步骤，则自动走异步会话模式。

### 6.2 前端通过 SSE 订阅执行事件

推荐第一阶段使用 **SSE（Server-Sent Events）**，而不是直接上 WebSocket。

原因：

- 前端实现简单
- 后端实现简单
- 天然适合“服务端持续推送日志和状态”
- 当前场景主要是单向推送，不急需双向协议

建议新增类似接口：

- `GET /api/execution-runs/{runId}/events/stream`
- `GET /api/execution-steps/{stepId}/events/stream`

前端进入详情页后：

1. 拉一次详情快照
2. 如果 run 还在执行，则建立 SSE 订阅
3. 收到事件后增量更新页面
4. 页面关闭或 run 结束时断开订阅

### 6.3 backend 统一做事件汇聚层

无论底层是：

- `code-processing`
- `AGENT_RUNTIME`
- 本地 CLI
- 未来其他 Runner

都统一向 backend 上报标准化事件。

backend 负责：

- 事件持久化
- 当前步骤状态聚合
- 最新日志尾部缓存
- 进度计算
- 对前端输出统一 SSE

这会让 frontend 与执行器解耦，也让后续新增执行器更容易。

## 7. 超时模型改造

### 7.1 不再使用单一 timeout 覆盖所有阶段

建议把当前 timeout 拆分为四类：

### 1. 提交超时

定义：backend 向执行器提交“启动步骤”的请求时，最多等多久。

建议范围：

- `5s ~ 30s`

用途：

- 避免执行器根本没响应时请求一直挂住

### 2. 心跳超时

定义：步骤开始后，超过多久没有任何事件、日志或 heartbeat，就判定 runner 失联。

建议范围：

- `30s ~ 120s`

用途：

- 判断“卡死”而不是“还在正常跑”

### 3. 空闲超时

定义：在步骤运行中，如果长时间没有 stdout/stderr、也没有状态更新，判定为异常空闲。

建议范围：

- `2min ~ 10min`

用途：

- 防止 CLI 进程挂起但不退出

### 4. 步骤最大执行时长

定义：步骤整体允许的最长运行时间。

建议按步骤类型区分：

- `PLAN`: 10 分钟
- `IMPLEMENT`: 30 到 60 分钟
- `TEST`: 20 到 40 分钟
- `REPORT`: 5 到 10 分钟

### 7.2 Agent 配置层需要从单 timeout 升级为 timeout profile

当前 Agent 只配置一个 `timeoutSeconds`，未来建议扩展为：

- `submitTimeoutSeconds`
- `heartbeatTimeoutSeconds`
- `idleTimeoutSeconds`
- `maxRuntimeSeconds`

第一阶段可以兼容旧字段：

- 旧的 `timeoutSeconds` 继续保留
- 新字段为空时，按 accessType 和 stepCode 走默认策略

### 7.3 300 秒限制应从“硬编码上限”改为“同步模式上限”

保留一个结论：

- **同步调用仍可保留 300 秒上限**
- **异步长任务模式不应继续受这个上限约束**

这样可以兼顾：

- 简单任务不复杂化
- 长任务不再被同步请求模型卡死

## 8. 日志与输出模型改造

### 8.1 输出分三层

建议把步骤输出拆成三层：

### 1. 结构化摘要层

保存最重要的信息：

- 当前命令
- 最近状态
- 步骤摘要
- 错误摘要
- 关键统计

这个层级用于列表页、步骤卡片和顶部摘要。

### 2. 实时尾日志层

保存最近 N 行或最近若干 KB 文本，用于前端实时查看。

特点：

- 只保留窗口
- 适合持续刷新
- 不会把所有历史日志都堆到前端

### 3. 完整原始日志层

完整 stdout/stderr 不建议继续直接塞进数据库长文本主字段，而应落到：

- 本地文件
- MinIO
- 或单独的日志分片表

前端默认不全量拉取，只在以下场景才读取：

- 用户点击“查看完整日志”
- 用户点击“下载日志”
- 排障时按需展开

### 8.2 推荐的事件类型

建议标准化事件类型：

- `step_started`
- `command_started`
- `stdout_chunk`
- `stderr_chunk`
- `heartbeat`
- `progress_changed`
- `command_finished`
- `artifact_ready`
- `step_summary_updated`
- `step_finished`

### 8.3 重复日志和超长日志需要压缩

对于以下日志，不应原样持续刷屏：

- 安装进度
- 重复 warning
- 重复轮询输出

建议支持：

- 重复行合并计数
- 进度行覆盖更新
- 尾日志窗口裁剪

例如：

- `WARN ... repeated 287 times`
- 只保留最后一次 `pnpm progress`

---

## 9. CLI 响应式输出方案

### 9.1 code-processing / runtime 内部 runner 需要以流式方式读取子进程输出

不再等待 `subprocess.run(...)` 结束后一次拿到完整 stdout/stderr，而应改为：

1. `Popen`
2. 按行或按 chunk 读取 stdout/stderr
3. 实时发出事件
4. 进程退出后发送最终结果

这一步是“响应式输出”的基础。

### 9.2 progress 不能只靠百分比推测

CLI 场景里的进度应分两类：

### 可计算进度

例如：

- 多仓按仓推进
- `PLAN -> IMPLEMENT -> TEST -> REPORT`
- 多命令顺序执行

这类仍然可以保留 `progressPercent`。

### 不可计算进度

例如：

- `pnpm install`
- `vite build`
- `mvn test`
- LLM CLI 推理

这类更重要的是：

- 当前命令
- 最近日志
- 是否仍在输出 heartbeat

因此页面不应强依赖百分比，而应同时展示“运行信号”。

### 9.3 页面展示建议

执行详情页建议升级为三块：

### 步骤概览卡

展示：

- 状态
- 当前命令
- 最近更新时间
- 简短摘要

### 实时日志面板

展示：

- 最近 100 到 500 行
- 自动滚动
- 支持暂停跟随
- 支持筛选 stdout / stderr

### 完整产物与完整日志

展示：

- 结构化 JSON 结果
- Markdown 报告
- 完整日志下载

这样用户既能“盯过程”，又不会因为整页大文本而卡死。

---

## 10. 数据模型建议

### 10.1 新增步骤事件表

建议新增 `execution_step_event`，用于持久化事件流。

建议字段：

- `id`
- `step_id`
- `run_id`
- `event_type`
- `sequence_no`
- `payload_json`
- `created_at`

用途：

- 页面刷新后可恢复最近事件
- SSE 断线后可从 `lastEventId` 继续追
- 审计和排障有据可查

### 10.2 步骤实体保留聚合字段

现有 `ExecutionStepEntity` 仍应保留聚合后的可见字段：

- `status`
- `progressPercent`
- `latestMessage`
- `outputSnapshot`
- `errorMessage`

但这些字段不再承担“完整日志存储”的职责，而应仅作为：

- 当前状态快照
- 页面首屏摘要

### 10.3 完整日志建议做对象化存储

建议把完整日志落为 artifact 或对象文件，例如：

- `STEP_RAW_LOG`
- `STEP_STDOUT_LOG`
- `STEP_STDERR_LOG`

数据库中只保留：

- `objectKey`
- `title`
- `previewText`

而不是直接把几十 MB 文本塞进主表。

## 11. API 与协议建议

### 11.1 执行器启动接口

长任务执行器建议返回：

```json
{
  "sessionId": "step-123-session-abc",
  "accepted": true,
  "runnerType": "CLI",
  "workspaceRoot": "...",
  "startedAt": "2026-04-18T21:40:10Z"
}
```

### 11.2 执行器事件上报接口

执行器向 backend 上报：

```json
{
  "sessionId": "step-123-session-abc",
  "sequenceNo": 17,
  "eventType": "stdout_chunk",
  "payload": {
    "text": "pnpm install ...",
    "command": "pnpm install --frozen-lockfile --prefer-offline"
  }
}
```

### 11.3 前端 SSE 输出格式

backend 对前端输出：

```text
event: execution-step-event
id: 17
data: {...}
```

前端根据事件类型增量更新：

- 当前步骤
- 当前命令
- 尾日志
- 进度状态

---

## 12. 前端改造建议

### 12.1 默认只渲染摘要和尾日志

详情页首屏建议只展示：

- 当前运行状态
- 步骤列表
- 每步最新摘要
- 每步最近日志窗口

不要默认渲染：

- 全量 `outputSnapshot`
- 全量超长 artifact 文本

### 12.2 超长内容延迟加载

建议引入：

- “查看完整日志”
- “下载完整日志”
- “仅显示最后 200 行”

当用户不主动点开时，不加载完整内容。

### 12.3 日志面板使用虚拟列表

如果前端要显示实时尾日志，建议用虚拟滚动或行级裁剪，避免日志越多 DOM 越大。

### 12.4 实时状态建议增加字段

除了 `progressPercent`，还建议前端展示：

- 当前命令
- 最近输出时间
- 日志速率或 heartbeat 状态

这样用户能分辨：

- 仍在运行
- 暂时安静
- 可能卡住

---

## 13. 分阶段落地建议

## 第一阶段：异步会话与超时解耦

目标：

- 解除 CLI 长任务对 300 秒同步请求的依赖

内容：

- 引入步骤执行会话
- 拆分提交超时、心跳超时、最大运行时长
- backend 不再阻塞等待完整 CLI 结果

## 第二阶段：日志分层与完整日志对象化

目标：

- 避免大段文本直接塞到页面和主表

内容：

- 新增步骤事件表或日志分片表
- 完整日志落 artifact / 对象存储
- 页面默认只展示摘要和尾日志

## 第三阶段：SSE 实时推送

目标：

- 实现响应式 CLI 输出

内容：

- frontend 建立 SSE
- backend 聚合执行器事件
- 页面实时刷新最新命令、最新日志、最新状态

## 第四阶段：体验增强

目标：

- 提升可观测性和排障体验

内容：

- 重复日志折叠
- 断点续传
- 搜索日志
- 错误高亮
- 失败定位卡片

---

## 14. 验收标准

改造完成后，至少应满足：

- `PLAN`、`IMPLEMENT`、`TEST` 不再受同步 HTTP 300 秒整体限制
- CLI 步骤运行中，前端可实时看到最新输出
- 页面默认不再全量渲染超长 stdout/stderr
- 用户可以查看最近日志，也可以按需下载完整日志
- 步骤执行过程断线后，页面刷新仍可恢复最近状态和日志尾部
- 现有 artifact、步骤状态、报告产物机制仍然可用

---

## 15. 风险与注意事项

### 15.1 事件量过大

如果每一行日志都持久化为一条事件，数据库压力可能明显上升。

建议：

- 按 chunk 合并
- 对重复输出做压缩
- 尾日志与完整日志分层

### 15.2 SSE 重连与幂等

前端断线重连后，backend 需要支持从 `lastEventId` 继续拉取，避免重复展示或丢日志。

### 15.3 执行器上报失败

如果 runner 正常执行，但事件上报失败，backend 需要：

- 基于 heartbeat 判定状态
- 最终仍能通过收尾接口提交结构化结果

### 15.4 历史任务兼容

历史任务不要求立刻拥有实时流式能力。

建议策略：

- 老任务继续走快照模式
- 新任务或新运行批次优先启用流式模式

---

## 16. 推荐结论

这次改造的本质，不是简单把某个 timeout 从 `300` 改成更大，而是把执行中心从：

- **同步调用型页面**

升级为：

- **异步任务执行系统 + 流式日志系统**

推荐优先级如下：

1. **先拆超时模型，解除 300 秒同步约束**
2. **再做日志分层，避免全量输出把页面拖垮**
3. **最后通过 SSE 做真正的响应式 CLI 输出**

按这个顺序推进，风险最小，也最容易逐步验证收益。
